# 场景内移动同步设计

本文描述 MyBevy 正式玩法场景中的移动同步方案。当前第一版已落地为 500x500 的机器人同步场景：两个客户端加入同一服务端房间后，各自启动一个本地机器人，并在两端通过 authority 帧回放看到一致的位置。

本文记录当前第一版实现状态和后续演进约定。已落地入口是 `arena.robot_sync`，玩法模块在 `project/src/game/features/robot_sync/`，MyServer room policy 为 `robot_sync_room`。

## 1. 目标范围

第一版只解决这些问题：

- 客户端进入一个正式 gameplay 场景，例如 `arena.robot_sync`。
- 两个客户端加入同一个 MyServer room。
- 每个客户端有一个本地输入源，默认由 bot 按规则生成移动输入，也可以切换为键盘手动控制。
- 移动输入通过 authority 发送，不直接改远端位置。
- 所有客户端消费同一批 `AuthorityEvent::FrameApplied`。
- 同一 `frame_id` 下，同一机器人在所有客户端得到相同位置。
- 机器人被限制在 500x500 场地内。

第一版不处理：

- 复杂物理碰撞。
- 寻路网格。
- 动画状态机。
- 技能、战斗和命中判定。
- 资源采集完整规则。
- 服务端直接广播坐标并覆盖客户端模拟。
- 大世界兴趣管理或区域同步。

这些能力后续仍应放在 `docs/gameplay/` 下拆分专题，而不是塞进场景框架文档。

### 1.1 Robot Sync 3D 表现升级边界

后续将把 `arena.robot_sync` 从当前 2D/UI 化表现升级为真实 3D 场景：地板、灯光、3D 摄像机和人物模型都属于客户端表现层改造。

本次 3D 表现升级明确不包含：

- 修改 MyServer 服务端代码。
- 修改 MyServer `robot_sync_room` policy。
- 修改 `project/src/framework/network/` 网络层。
- 修改 authority 协议、`robot_move` action 或 payload 字段。
- 修改 `RobotSyncReplayState` 的同步语义。

实现时应优先复用现有 `AuthorityEvent::FrameApplied` 回放结果，把 fixed 坐标转换为 3D 场景中的 `Transform`。开发阶段只要求完成必要的客户端编译、格式化和单元测试检查；单机、LAN 和 MyServer 的运行验收由客户端表现层开发完成后统一手动执行。

## 2. 和现有系统的关系

移动同步属于 game layer 玩法规则。

相关边界如下：

- `project/src/framework/scene/`：负责进入/退出场景、加载资源、创建场景 root、相机和基础 anchor，不负责移动规则。
- `project/src/game/authority/`：提供统一控制机命令和事件，例如 `AuthorityCommand::SendInput`、`AuthorityEvent::FrameApplied`。
- `project/src/game/myserver/`：负责登录、连接、加房、ready、start 和协议适配。
- `project/src/game/features/robot_sync/`：承载机器人同步玩法，包括 authority 启动、bot 输入、帧回放、状态模拟、视觉实体和 HUD snapshot。
- `project/src/game/scenes/robot_sync_arena.rs`：只处理 `arena.robot_sync` 的场地内容实例化和场景进入响应。

原则是：场景提供空间和生命周期，玩法模块提供移动和同步规则。

## 3. 当前同步模型

机器人移动采用“输入同步 + 确定性回放”：

1. 本地客户端根据 player id 启动自己的输入源。
2. 输入源生成移动输入，例如方向和速度；默认输入源是 bot，也可以用 `ROBOT_SYNC_INPUT_MODE=manual` 切到键盘手动控制。
3. 客户端把输入发送给 authority，目标帧为 `session.frame_id + input_delay_frames`。
4. MyServer 或本地/LAN authority 按帧收集输入。
5. authority 下发 `AuthorityEvent::FrameApplied { frame }`。
6. 玩法系统只在收到权威帧后推进机器人状态。
7. 渲染系统把机器人状态同步到 Bevy entity transform。

这个模型的关键点是：本地输入不能直接决定最终位置，本地输入也必须等权威帧回来后才进入模拟。这样两个客户端只要拿到同一批帧和同一套模拟规则，就能得到一致位置。

## 4. 场景和坐标

机器人同步场景第一版当前配置：

- 场景 ID：`arena.robot_sync`
- 场地尺寸：500x500
- 逻辑坐标：`x/z = -250..250`
- 原点：场地中心
- 相机：顶视 2D 或 3D 正交相机
- 场地内容：地面、边界、网格、出生点、机器人

逻辑模拟和渲染应分层：

- 模拟层保存确定性坐标。第一版统一使用整数定点坐标，不使用 `f32` 作为权威逻辑位置。
- 渲染层把模拟坐标转换为 `Transform`。
- HUD 和日志只读取模拟层状态。

定点单位约定：

- `FIXED_UNIT = 1000`，即 1 个世界单位等于 1000 个 fixed units。
- 场地边界为 `x/y = -250000..250000` fixed units，其中逻辑 `y` 轴渲染到 Bevy 的 `z` 轴。
- 渲染转换为 `world_x = fixed_x / 1000.0`，`world_z = fixed_y / 1000.0`。
- 日志 telemetry、HUD 坐标和一致性检查全部使用 fixed units，不使用渲染后的浮点坐标。

## 5. 输入协议

第一版使用 generic player input 通道，action 名称固定为：

```text
action = "robot_move"
```

payload 使用 JSON object，并保持字段小而稳定。最终采用 camelCase 字段名：

```json
{
  "version": 1,
  "seq": 12,
  "botTick": 40,
  "dirX": 1000,
  "dirY": 0,
  "speed": 10000
}
```

字段含义：

- `version`：整数，当前只能为 `1`。
- `seq`：整数，本地输入序号，范围 `0..2147483647`，用于日志和重复输入诊断。
- `botTick`：整数，bot 内部 tick，范围 `0..2147483647`，用于复现输入生成。
- `dirX` / `dirY`：整数，方向向量定点值，范围 `-1000..1000`，`1000` 表示一个单位方向。逻辑 `dirY` 渲染到 Bevy `z` 轴。
- `speed`：整数，单位为 fixed units per second。当前客户端 replay 支持 `0..60000`；当前 MyServer `robot_sync_room` 校验范围是 `0..10000`，因此内置 bot 默认发送 `10000`，避免端到端联调被服务端拒绝。

payload 边界：

- JSON 必须是 UTF-8 object，最大长度 `256` bytes。
- 必须包含上述 6 个字段，字段名使用 camelCase；第一版服务端应拒绝未知字段，避免客户端和服务端对 payload 语义理解不一致。
- 所有字段必须是整数，拒绝字符串、浮点数、布尔值、数组、对象和 `null`。
- MyBevy replay 要求 `dirX * dirX + dirY * dirY <= 1000 * 1000`。例如水平移动使用 `{ "dirX": 1000, "dirY": 0 }`，对角线移动应使用约 `{ "dirX": 707, "dirY": 707 }`，不能发送 `{ "dirX": 1000, "dirY": 1000 }`。当前 MyServer `robot_sync_room` 只校验 `dirX` / `dirY` 单字段范围，不计算向量长度；客户端和 mock 场景仍应发送 replay 合法的方向。
- `{ "dirX": 0, "dirY": 0, "speed": 0 }` 表示显式停止；`speed > 0` 时方向不能为零向量。
- target frame 不放在 payload 中，仍由 authority input envelope 承载。payload 内的 `botTick` 只用于输入生成诊断，不作为权威帧号。

输入延迟帧数：

- 第一版固定 `input_delay_frames = 2`，复用 Touch Ripple “当前 authority frame 加延迟帧发送输入”的思路。
- 客户端发送目标帧为 `target_frame = current_authority_frame + input_delay_frames`。
- MyServer `robot_sync_room` policy 当前也配置 `input_delay_frames = 2`。如果后续服务端策略改成不同值，客户端应记录错误并以服务端策略为准，避免两端 target frame 计算不一致。
- bot 不应为同一个 `target_frame` 重复发送等价输入；确实产生重复输入时，回放规则按本文件“排序、重复帧和缺失输入规则”处理。

## 6. 机器人状态

玩法状态当前集中在 `RobotSyncReplayState` 等资源中，核心 replay 状态等价于：

```text
RobotSyncState
  active_scene_session
  last_applied_frame
  robots: player_id -> RobotState
```

每个机器人状态当前包含：

```text
RobotState
  player_id
  position
  direction
  velocity
  speed
  last_input_seq
  last_frame
  spawn_index
  color_index
  visual_entity
```

字段约定：

- `player_id`：authority 玩家 ID，作为机器人主键。
- `position`：`x/y` fixed units，逻辑 `y` 渲染到 Bevy `z`。
- `direction`：最近一次被权威帧接受的 `dirX/dirY`，范围同 payload。
- `velocity`：由 `direction * speed` 推导出的 fixed units per second，可缓存用于 HUD，不作为独立权威输入。
- `heading`：当前第一版 replay state 不保存 heading；checksum 中的 `heading_milli_degrees` 固定为 `0`。
- `speed`：最近一次被权威帧接受的速度。
- `last_input_seq`：最近一次采用的 payload `seq`。
- `last_frame`：该机器人最后一次完成模拟推进的 authority frame。
- `spawn_index` / `color_index`：由稳定玩家排序或 player id hash 得到，不能依赖本地加入顺序。

状态更新规则：

- 收到 `AuthorityEvent::Snapshot` 时建立或校正玩家列表。
- 收到 `AuthorityEvent::FrameApplied` 时处理该帧输入。
- 如果帧号小于或等于 `last_applied_frame`，忽略。
- 如果同一帧多个玩家输入，按稳定顺序处理，例如按 `player_id` 排序。
- 如果同一玩家同一帧有多个 `robot_move` 输入，选择 `seq` 最大的一条；`seq` 相同则选择原始 frame input 列表中最后一条，并记录诊断日志。
- 如果玩家没有输入但此前已经收到过输入，第一版保持上一方向和速度继续移动；如果此前从未收到输入，则保持静止。
- 每次推进后 clamp 到场地边界。

移动参数：

- replay 默认速度：`60000` fixed units per second。
- replay 最大速度：`60000` fixed units per second。
- 内置 bot 默认发送速度：`10000` fixed units per second。当前 MyServer `robot_sync_room` 只接受 `0..10000`，后续如放宽服务端校验，再同步调整 bot 默认值和联调文档。
- 场地边界：`x/y = -250000..250000` fixed units，clamp 包含边界端点。
- 初始方向：新机器人在收到第一条合法 `robot_move` 前方向为 `(0, 0)`、速度为 `0`，避免手动客户端刚进场时在按键前自动漂移。
- 无输入行为：已有输入后保持上一方向和速度；触边后位置被 clamp，但方向不自动反弹，除非后续权威帧输入改变方向。
- 显式停止：收到合法零方向且 `speed = 0` 后停在当前位置，后续缺失输入继续保持停止。
- 出生点：第一版至少提供 4 个固定出生点。前两个玩家默认都在 `(0, 0)`，方便双客户端手动验收时直接观察同步；后续备用点为 `(-200000, 200000)`、`(200000, -200000)`。更多玩家按列表循环并用 `spawn_index` 区分。

## 7. 确定性推进

位置推进不使用 Bevy `Time` 的 delta。推进步长来自 authority frame：

```text
dt = 1 / frame.fps
```

整数定点推进公式固定为：

```text
denom = 1000 * frame.fps
delta_x = trunc_toward_zero(dirX * speed / denom)
delta_y = trunc_toward_zero(dirY * speed / denom)
position = clamp(position + delta, bounds)
```

`frame.fps` 必须大于 0。当前 MyServer `robot_sync_room` 使用 `active_room_fps = 20`。replay 默认速度 `60000` 在水平满速方向下每帧移动 `3000` fixed units，也就是 3 个世界单位；内置 bot 当前发送速度 `10000`，在 20 fps 下每帧移动 `500` fixed units。

确定性要求：

- 同一输入得到同一方向向量。
- 同一速度单位得到同一位移。
- 同一帧只推进一次。
- 排序规则固定。
- 边界 clamp 规则固定。
- 位置日志和 checksum 使用模拟层坐标，不使用渲染 transform。

排序、重复帧和缺失输入规则：

- 客户端只消费 `AuthorityEvent::FrameApplied`，不在本地 render/update tick 中推进逻辑位置。
- 全局 `last_applied_frame` 初始为空；收到第一帧后从该帧建立基准并推进一次。后续只接受 `frame_id > last_applied_frame` 的帧。
- `frame_id <= last_applied_frame` 的重复或乱序帧必须忽略，并输出包含 `frame_id`、`last_applied_frame` 和 `room_id` 的诊断日志。
- 如果 authority 连续下发帧但某个玩家缺少输入，该玩家按“无输入行为”推进；不能因为缺失输入暂停整个场景。
- 如果 authority 帧中存在未知 action，机器人同步模块忽略该输入并记录 debug 级日志，不影响同帧其他合法输入。
- 如果 payload JSON 解析失败或字段越界，客户端忽略该条输入并记录错误；服务端第一版应在入房输入阶段尽量拒绝这类 payload，使非法输入不进入 frame bundle。

不要在权威帧之外用本地帧率补偿机器人逻辑位置。视觉上需要平滑时，可以额外做插值实体，但插值结果不能反写模拟状态。

## 8. MyServer 接入

MyServer 在这个方案中只负责远端 authority endpoint 和房间帧分发。

当前流程：

1. `AuthorityCommand::Join { endpoint: AuthorityEndpoint::MyServer { ... } }`
2. `MyServerCommand::GuestLogin { connect_game: true }`
3. `MyServerCommand::JoinRoom { room_id, policy_id: "robot_sync_room" }`
4. `MyServerCommand::SetReady { ready: true }`
5. `MyServerCommand::StartRoom`
6. `AuthorityCommand::SendInput { action: "robot_move", ... }`
7. `AuthorityEvent::FrameApplied { frame }`

服务端策略第一版复用 generic `SendPlayerInput` 和 `FrameBundlePush`。MyServer 已新增显式 `robot_sync_room` policy 和 `RobotSyncRoomLogic`，不会依赖未知 policy 回退。`robot_sync_room` 当前关键参数：

- `max_members = 32`
- `min_start_players = 1`
- `allow_join_in_game = true`
- `active_room_fps = 20`
- `input_delay_frames = 2`
- `wait_strategy = Optimistic`
- `missing_input_strategy = Empty`
- `movement_correction_interval_frames = 0`
- `movement_aoi_enabled = false`
- `movement_control_stop_frames = 0`

MyServer 对 `robot_move` 的职责只包括：

- 校验 action 必须等于 `"robot_move"`。
- 校验 payload 是合法 JSON object，长度、字段、整数范围、方向字段范围和速度范围符合本文协议。当前方向向量长度由 MyBevy replay 侧校验。
- 接受合法输入并按 authority 帧通道转发。
- 拒绝非法 action、非法 JSON、超大 payload、非整数字段、方向越界、速度越界和版本不匹配。

当前服务端错误码包括：

- `INVALID_ROBOT_MOVE_ACTION`
- `INVALID_ROBOT_MOVE_JSON`
- `ROBOT_MOVE_PAYLOAD_TOO_LARGE`
- `ROBOT_MOVE_PAYLOAD_NOT_OBJECT`
- `ROBOT_MOVE_FIELDS_MISMATCH`
- `ROBOT_MOVE_FIELD_NOT_INTEGER`
- `INVALID_ROBOT_MOVE_VERSION`
- `UNSUPPORTED_ROBOT_MOVE_VERSION`
- `ROBOT_MOVE_SEQ_OUT_OF_RANGE`
- `ROBOT_MOVE_BOT_TICK_OUT_OF_RANGE`
- `ROBOT_MOVE_DIR_OUT_OF_RANGE`
- `ROBOT_MOVE_SPEED_OUT_OF_RANGE`

服务端第一版明确不做这些事情：

- 不计算机器人位置。
- 不广播机器人坐标。
- 不做碰撞、寻路、边界 clamp 或速度修正。
- 不发送 `MovementSnapshotPush` 覆盖客户端机器人状态。
- 不根据服务端 tick 改写 payload 中的方向、速度或 bot tick。

只有服务端已经有强约束的移动快照策略时，才评估接入 `SendMoveInput`、`MovementSnapshotPush` 和 `MovementRejectPush`。

如果使用服务端移动快照，设计需要重新明确：

- 服务端是否直接计算位置。
- 客户端预测是否允许。
- 被拒绝输入如何回滚。
- snapshot 和本地模拟的优先级。

第一版机器人同步不建议上来就做回滚预测；先保证权威帧回放稳定。

MyBevy 双客户端脚本：

- `scripts/start-robot-sync-two-clients.ps1` 默认 MyServer 模式，写入 `MYBEVY_START_SCENE=arena.robot_sync`、同一 `AUTHORITY_MYSERVER_ROOM`、默认 `AUTHORITY_MYSERVER_POLICY=robot_sync_room`、不同 `AUTHORITY_PLAYER_ID` / `AUTHORITY_MYSERVER_GUEST_ID`。脚本默认 `-InputModeA manual -InputModeB off`，便于 A 端手动控制、B 端静止观察；需要自动 bot 时可显式传 `-InputModeA bot` 或 `-InputModeB bot`。脚本只设置 `ROBOT_SYNC_AUTHORITY_MODE`，并在 client launcher 中清空继承的 `AUTHORITY_DEV_MODE`，避免全局 dev authority 和场景内 authority 重复启动。
- MyServer mode 下，`-HostAddress` 写入 `MYSERVER_GAME_HOST`；`-Transport tcp` 时 `-Port` 写入 `MYSERVER_TCP_FALLBACK_PORT`，`-Transport kcp` 时写入 `MYSERVER_KCP_PORT`。
- 当前本地 MyServer 完整栈需要在 `C:\project\MyServer` 执行 `dev-stack.ps1 -WithMatch`。

## 9. 本地输入生成

本地输入源不是权威对象。它只负责产生自己玩家的输入，并通过 authority 帧通道发送。

当前规则：

- seed 来自 player id、room id 或显式环境变量。
- 输入源在 `arena.robot_sync` active 且存在本地 authority player 时运行；MyServer 连接未打开时早期输入可能记录一次非阻塞 warning，后续连接打开后继续发送。
- 输入源不直接修改 `RobotState.position`。
- 输入源按固定 tick 或每 N 个 authority frame 发送一次输入。
- 发送目标帧使用 `session.frame_id + input_delay_frames`。
- 同一个 target frame 不重复发送等价输入。

当前第一版 bot 使用简单方向序列：

- seed 来自 player id 的 FNV-1a 32-bit。
- 每 20 个 bot tick 切换一次方向。
- 方向集合包含水平、垂直和 707 对角线方向，保证方向长度不超过 1000。
- 只记录 `last_sent_target_frame`，避免同一 target frame 重复发送。

手动输入模式：

- 设置 `ROBOT_SYNC_INPUT_MODE=manual` 后，本地输入源改为键盘控制。
- `WASD` 和方向键控制本地机器人方向；同时按水平和垂直方向时发送 707 对角线方向。
- 按键按下时持续发送当前方向输入；松开全部移动键时发送一次 `{ "dirX": 0, "dirY": 0, "speed": 0 }` 停止输入。
- 手动输入默认速度为 `ROBOT_SYNC_MANUAL_SPEED=10000`，保持在当前 MyServer `robot_sync_room` 校验范围内。

这些规则必须只影响输入生成，不影响远端机器人的本地模拟规则。

## 10. Late Join 和 Snapshot

如果第二个客户端晚加入，必须能拿到已有玩家列表和初始状态。

第一版当前采用低成本策略：

- `AuthorityEvent::Snapshot.players` 用于创建机器人占位。
- 所有机器人使用稳定出生点。
- 如果 snapshot 的玩家集合没有变化，保留已有机器人运动状态。
- 如果 snapshot 的玩家集合发生变化，重建 replay 基线：按排序后的玩家列表重新分配 spawn/color，`last_applied_frame` 在已有基线时不回退。这样处理 late join 时，两端会以同一玩家集合和出生点重新对齐后续 replay。

这仍不等同于服务端携带历史位置快照；晚加入客户端不会恢复加入前完整轨迹。如果要求晚加入后立刻看到准确历史位置，则服务端或 authority snapshot 需要携带机器人位置状态：

```json
{
  "robots": [
    {
      "playerId": "player-a",
      "frameId": 120,
      "x": 10240,
      "y": -5000,
      "dirX": 1000,
      "dirY": 0
    }
  ]
}
```

是否实现机器人 snapshot 应作为独立任务评估，不和最小双客户端同步闭环混在一起。

## 11. 清理边界

场景退出时必须清理：

- 机器人 Bevy entity。
- `RobotSyncState` 中的 scene session、robots 和 last frame。
- bot 本地状态。
- MyServer join 跟随状态。
- HUD 状态。

不应清理的内容：

- 全局登录态。
- MyServer 配置资源。
- authority 插件本身。
- 场景 catalog 和 manifest 注册。

是否在退出机器人场景时发送 `AuthorityCommand::Leave` 需要按玩法模式决定。若该 authority 只服务当前场景，应离开；若后续要跨场景保持同一 room，需要保留连接但停止消费旧场景输入。

## 12. 一致性校验

HUD 和日志 telemetry 合起来应能回答三个问题：

- 当前应用到第几帧。
- 当前本地玩家是谁。
- 每个机器人在该帧的位置是什么。

当前客户端 telemetry 输出为 `robot sync frame applied`：

```text
frame_id=120 robot_count=2 checksum=9f1d69aa robots=player-a:x=10240,y=-5000,last_frame=120;player-b:x=-20000,y=4000,last_frame=120
```

坐标日志规则：

- `x` / `y` 输出 fixed units 原值。
- 如需给人读的坐标，可额外输出四舍五入到 3 位小数的世界坐标，例如 `worldX=10.240 worldY=-5.000`，但不能用它参与 checksum。
- 每个客户端同一 `frame_id` 下同一 `player_id` 的 fixed 坐标必须一致。

客户端关键日志字段：

- 启动 / 加房：`player_id`、`guest_id`、`room_id`、`policy_id`、transport、host/port、错误码和原因。
- 本地输入：`player_id`、`target_frame`、`seq`、`botTick`、`dirX`、`dirY`、`speed`，自动 bot 日志为 `sending robot sync bot input`，手动键盘日志为 `sending robot sync manual input`。
- authority replay：`FrameApplied.frame_id`、`fps`、输入 action、payload 解析拒绝原因、重复/乱序 frame、zero fps。
- telemetry：`frame_id`、`robot_count`、`checksum`、`robots`。

服务端关键日志或 mock-client 输出字段：

- `room_id`、`policy_id`、`player_id`、`guest_id`。
- `frame_id`、`action=robot_move`、`input_delay_frames`、`FrameBundlePush.fps`。
- `PlayerInputRes.ok`、`errorCode`、payload 校验错误码。
- `RobotSyncRoomLogic.get_serialized_state()` 中的 `logicType=robot_sync_room`、`tickCount`、`lastFrame`、`recentInputs`。

checksum 第一版使用确定性 32-bit 整数规则，不依赖平台默认 hash：

```text
robot_hash = fnv1a32(player_id_utf8)
robot_value = robot_hash ^ low32(x) ^ rotl32(low32(y), 1) ^ rotl32(low32(heading_milli_degrees), 7) ^ low32(last_frame)
checksum = wrapping_sum(robot_value 按 player_id 字典序遍历所有机器人)
checksum = checksum ^ low32(frame_id)
```

其中：

- `fnv1a32` 使用 FNV-1a 32-bit，offset basis `2166136261`，prime `16777619`。
- `low32` 表示把有符号 fixed 坐标按二进制补码取低 32 位。
- `heading_milli_degrees` 当前第一版固定为 `0`。后续如果实现朝向角，必须同步更新 replay state、telemetry checksum 和双客户端验收脚本；如果 HUD 也展示朝向或 checksum，再同步更新 HUD snapshot。
- `wrapping_sum` 使用 32-bit wrapping add。
- checksum 输入必须按 `player_id` 字典序排序，不能按 HashMap 迭代顺序。

验收时两个客户端在同一 frame 的 checksum 应一致。如果不一致，优先检查：

- 两端是否消费了同一批 `FrameApplied`。
- 输入排序是否一致。
- 是否有本地 `Time` 修改了模拟位置。
- 是否存在重复 frame 或乱序 frame 被重复应用。
- spawn 点和初始状态是否一致。

双客户端排障顺序：

1. 未进房：检查 MyServer 是否用 `dev-stack.ps1 -WithMatch` 启动，客户端 `MYSERVER_GAME_HOST`、`MYSERVER_TCP_FALLBACK_PORT` / `MYSERVER_KCP_PORT` 是否指向 `game-proxy` 入口，guest id 是否不同。
2. 错误 policy：检查客户端 `AUTHORITY_MYSERVER_POLICY` / `ROBOT_SYNC_MYSERVER_POLICY` 是否为 `robot_sync_room`，服务端日志是否显示 `policy_id=robot_sync_room`，不要依赖未知 policy 回退。
3. 输入被拒：查看 `PlayerInputRes.errorCode`，按 `INVALID_ROBOT_MOVE_ACTION`、`INVALID_ROBOT_MOVE_JSON`、`ROBOT_MOVE_FIELDS_MISMATCH`、`ROBOT_MOVE_DIR_OUT_OF_RANGE`、`ROBOT_MOVE_SPEED_OUT_OF_RANGE` 等错误码回查 payload。
4. 未收到 `FrameApplied`：检查房间是否 ready/start 成功、`input_delay_frames=2`、target frame 是否超过窗口、客户端是否在 game connection open 前过早发送输入。
5. checksum 不一致：按同一 `frame_id` 对比两端 `robots` 字符串，确认玩家集合 snapshot 是否一致、是否有 zero fps/重复帧、是否有非法 payload 被一端忽略。
6. 退出清理异常：返回大厅后检查是否仍有机器人实体、HUD、bot/replay/visual 状态或旧 authority session；日志应出现 `robot sync authority cleanup`。

## 13. 后续扩展方向

这个目录后续应继续承载这些 gameplay 能力：

- 角色行为状态机：idle、move、interact、gather、cast、stun 等。
- 采集和交互：采集目标、进度、取消、结果同步。
- 寻路：点击目标点、路径请求、路径跟随、阻挡处理。
- AI/机器人：行为树或轻量状态机、巡逻、追踪、避障。
- 动作同步：动作开始帧、持续帧、取消窗口、冷却。
- 快照和回放：late join、断线重连、观战和一致性校验。

这些能力可以共享 actor identity、frame、action payload、snapshot 和 deterministic simulation 的基础约定，但每个专题应单独记录规则，避免把所有玩法行为堆进一个移动同步模块。
