# 方圆灵构渲染技术路线

## 文档定位

本文记录方圆灵构体系下，后续整个游戏功能的渲染技术路线和阶段开发路线。

本文不是某个单点功能的实现说明，而是用于约束后续家园、装备、技能、NPC、天道自演、场景道具、宗门工程、多人热点和纪元长期内容的渲染方向。

本文基于这些前提：

- 当前项目使用 Rust stable 和 Bevy `0.18.1`。
- 当前世界观采用赛博修仙和方圆二相。
- 游戏内可生成、可继承、可审核的视觉内容，应尽量由正方体和球体组合而成。
- 当前第一阶段家园原型通过本地 RON 蓝图生成静态方块和球体预览。
- 从初始设计开始，游戏 Entity 的粒度就是角色、技能实例、家园物件、装备物件、天道生成物等逻辑对象；方圆 primitive 是这些逻辑对象内部的渲染数据，不是玩法层 Entity。
- 后续长期功能需要覆盖家园、装备、技能、NPC、场景装饰、天道自演和多人热点场景。
- 长期目标是符合世界观并支撑持续扩展，而不是只让当前 demo 能跑。

本文与这些文档相关：

- [方圆灵构](./方圆灵构.md)：定义方相、圆相、灵构蓝图、灵构额度和 AI 生成边界。
- [方圆灵构蓝图规则](./方圆灵构蓝图规则.md)：定义第一阶段本地家园蓝图格式和生成限制。
- [方圆灵构程序动画设计](./方圆灵构程序动画设计.md)：定义程序动画、技能表现、同步边界和性能预算。
- [天道自演](./天道自演.md)：定义世界主动生成内容的分层、生命周期和审核边界。
- [天道灵压](./天道灵压.md)：定义热点区域分流、特效降级和技术限压的世界观包装。
- [玩法体系设想](./玩法体系设想.md)：定义方圆洞府、灵构工坊、试炼场、宗门工程和纪元玩法池。

## 核心结论

方圆灵构不适合走 Minecraft 式 Chunk Meshing 作为长期主路线。

Minecraft 式 Chunk Meshing 适合大量规则网格方块构成的连续地形。当前项目的核心限制不同：

- 基础可见元素只有方相和圆相。
- primitive 可以有不同位置和大小。
- 不存在旋转。
- 颜色、材质、发光和透明需要多样化。
- 家园、装备、技能、NPC 和天道自演都需要复用同一种视觉语法。
- 内容需要由 AI 或系统生成，并经过审核、预算和降级。

因此长期路线应是：

```text
方圆 primitive 数据化
-> 统一蓝图审核和预算
-> 统一 runtime primitive 表达
-> 静态实例池和动态 VFX 实例池
-> cube / sphere 两条渲染批
-> 材质参数实例化
-> LOD、AOI 和天道灵压降级
```

简化口径：

```text
世界观看起来是万物方圆
技术底层是少量 mesh + 海量实例数据
```

## 总体原则

### primitive 从一开始就不是玩法 Entity

本路线明确采用：

```text
Entity = 角色、技能、家园物件、装备物件、天道生成物、场景交互物
Primitive = 逻辑 Entity 内部的方圆渲染数据
```

也就是说，正式架构从第一阶段开始就不把一个方块或一个球体设计成玩法 Entity。

推荐初始结构：

```text
CharacterEntity
  CharacterState
  EquipmentVisualSet

SkillInstanceEntity
  SkillRuleState
  VfxRuntime

HomeObjectEntity
  HomeObjectState
  FangyuanPrimitiveSet

TiandaoManifestationEntity
  ManifestationLifecycle
  FangyuanPrimitiveSet
```

渲染适配层可以在内部生成批量 mesh、实例数据或渲染 draw item，但这些都不应暴露为玩法层 primitive Entity。

危险形态：

```text
一个方块 = 一个玩法 Entity
一个球体 = 一个玩法 Entity
一个技能里 100 个 primitive = 100 个动态玩法 Entity
1000 个玩家同时释放技能 = 100000 个动态玩法 Entity
```

这个结构会让 CPU 端 ECS 遍历、Transform 更新、可见性计算、Extract 阶段和渲染世界同步成为瓶颈。

目标结构应是：

```text
一个角色 = 一个角色逻辑 Entity
一个家园物件 = 一个家园物件逻辑 Entity
一个技能表现 = 一个 VFX 运行时实例
一个天道生成物 = 一个生命周期对象
这些对象内部持有 primitive 数组
渲染层按 cube / sphere 批量提交实例数据
```

### mesh 极少，实例极多

全局基础 mesh 应尽量少：

- unit cube。
- low sphere。
- medium sphere。
- high sphere。

cube 不需要多套精度。sphere 需要按平台和距离切换精度：

- 近处高表现：中高精度球。
- 普通场景：中精度球。
- 远处、移动端、热点区域：低精度球。
- 极端热点：可退化为 billboard、点光感或规则轮廓。

不建议早期引入太多基础形状。圆柱、圆环、文字、自由 mesh、粒子系统和玩家自定义 shader 都会增加审核和渲染复杂度，削弱方圆二相的统一风格。

### 材质合一优先于材质自由

颜色和材质多样化不能通过创建大量 `StandardMaterial` 来解决。

如果每种颜色、每种发光、每种粗糙度都创建一个独立材质，渲染批次会被切碎。长期应尽量让实例共享同一套方圆材质，通过实例数据表达差异。

推荐方向：

```text
同一个 cube mesh
同一个 sphere mesh
同一个或少量 FangyuanMaterial
每个实例携带 color / emissive / alpha / material_profile_id / texture_index
```

这样可以让视觉上多彩、多材质，底层仍保持高批处理效率。

### 颜色先行，质感后置

渲染路线应分阶段增加表现力：

1. 第一阶段只支持颜色。
2. 第二阶段支持发光强度和 alpha。
3. 第三阶段支持材质 profile，例如玉石、金属、灵木、石质、火核、水膜。
4. 第四阶段再考虑 texture atlas 或 texture array。
5. 最后才考虑更复杂的屏幕空间效果、溶解、噪声和特殊边缘光。

不要一开始就把材质系统设计成完整 PBR 资产管线。当前世界观的优势是抽象几何和程序表现，不是传统美术资产堆叠。

### 透明和动态光是预算资源

大量透明排序、半透明叠加、动态阴影和动态点光源会比 primitive 数量更早击穿性能。

默认规则：

- 静态家园和场景道具默认不透明。
- 透明只用于护盾、水膜、残影、淡出和特殊预警。
- 动态光源数量必须严格限制。
- 发光优先使用 emissive + bloom，而不是大量真实灯光。
- 热点区域优先降级透明和发光，再降低普通不透明 primitive。

世界观包装：

```text
区域灵压升高
-> 非关键灵光被天道压制
-> 装饰层显化变弱
-> 远处灵构只保留规则轮廓
```

### 服务端只同步语义事件

服务端不应模拟和同步每个视觉 primitive。

服务端关心：

```text
caster_id
skill_template_id
visual_blueprint_id
start_tick
position
direction 或 target
seed
```

客户端根据这些语义事件、本地缓存的视觉蓝图和确定性 seed 还原表现。

禁止长期依赖：

```text
同步每个 primitive 的每帧 position
同步每个 primitive 的每帧 scale
同步每个 primitive 的每帧 alpha
同步视觉播放器内部状态
```

### 规则层和个性层必须分离

技能、陷阱、护盾、范围效果等战斗表现必须分层。

规则层：

- 真实作用范围。
- 投射物方向和速度。
- 护盾边界。
- 陷阱触发区域。
- 危险提示。
- 友敌可读性。

个性层：

- 颜色倾向。
- 方相和圆相比例。
- 拖尾形态。
- 显化节奏。
- 命中扩散。
- 残留余辉。
- 装饰 primitive 分布。

玩家蓝图主要影响个性层。规则层由系统模板控制，不能被玩家隐藏、缩小、伪装或扩大。

## 推荐架构

### 总体数据流

推荐长期数据流：

```text
玩家输入 / 天道触发 / 系统模板
-> AI 或规则生成蓝图
-> 蓝图审核
-> 预算计算
-> 蓝图编译
-> runtime primitive set
-> 绑定到角色 / 技能 / 家园物件 / 天道生成物等逻辑 Entity
-> 静态实例池 / 动态 VFX 实例池
-> 渲染提交
-> LOD / 热点降级 / 调试统计
```

其中 AI 只能生成表现蓝图，不能生成最终强度。强度来自系统模板、角色权限、四属性、职业、称号、世界层级和服务端规则。

### 模块拆分建议

长期可以考虑这些模块边界：

```text
framework/fangyuan/
  blueprint
  validator
  compiler
  runtime_primitive
  material_profile
  budget

framework/rendering/
  primitive_meshes
  instance_buffer
  fangyuan_material
  lod
  visibility
  debug

framework/vfx/
  recipe
  curves
  player
  deterministic_seed
  vfx_instance_pool

game/homes/
  home_blueprint
  home_preview
  home_runtime

game/equipment/
  equipment_blueprint
  equipment_socket
  equipment_visual_runtime

game/skills/
  skill_template
  skill_visual_blueprint
  skill_authority_event
  trial_field

game/world_events/
  tiandao_manifestation
  generated_decoration
  manifestation_lifecycle
```

这些名字不要求一次性落地，但应作为长期职责边界参考。

### 方圆 primitive 统一结构

后续不要让家园、装备、技能、NPC 各自定义一套几何结构。

建议收敛到统一的运行时表达：

```text
FangyuanPrimitive:
  kind: cube | sphere
  position: Vec3
  size: Vec3
  color: LinearRgba
  alpha: f32
  emissive: f32
  material_profile_id: MaterialProfileId
  role: PrimitiveRole
  flags: PrimitiveFlags
```

`role` 用于表达语义：

- structure：结构层。
- core：灵核。
- boundary：规则边界。
- warning：预警范围。
- trail：拖尾。
- impact：命中反馈。
- decoration：装饰。
- socket：装备挂点辅助。
- archive：纪元档案展示。

这个 role 对渲染、审核、LOD 和战斗可读性都很重要。

例如热点降级时可以优先丢弃：

```text
decoration -> trail -> impact -> 非关键 core
```

但应保留：

```text
warning -> boundary -> 规则 core
```

### 蓝图和 runtime 分离

蓝图适合存储、审核、继承和网络引用。

runtime primitive 适合渲染、LOD 和运行时播放。

不要把蓝图结构直接当运行时结构长期使用。蓝图需要可读、稳定、可版本化；runtime 需要紧凑、连续、适合批量处理。

推荐关系：

```text
BlueprintPrimitive
-> 审核
-> 编译
-> RuntimePrimitive
-> InstanceData
```

蓝图字段可以保留描述性信息：

- name。
- description。
- author。
- version。
- bounds。
- tags。
- budget。
- primitives。

runtime 字段应尽量紧凑：

- mesh_kind。
- position。
- scale。
- color。
- material_profile_id。
- packed flags。

## 渲染管线路线

### 第一阶段：逻辑 Entity + 标准渲染适配

第一阶段可以继续使用 Bevy 标准 mesh、材质和渲染资源，但 Entity 粒度仍然必须是逻辑对象，而不是 primitive。

推荐第一阶段结构：

- `HomeObjectEntity` 或 `FangyuanObjectEntity` 表示一个家园物件、装饰物、阵法节点或预览对象。
- `FangyuanPrimitiveSet` 保存该逻辑对象内部的 cube / sphere 数据。
- `Transform` 只表达逻辑对象整体位置，不为每个 primitive 创建 ECS Transform。
- 渲染适配层根据 `FangyuanPrimitiveSet` 生成一个或少量 `Mesh` / render batch / 简化 instance 数据。
- `Mesh3d`、`MeshMaterial3d<StandardMaterial>` 或项目当前材质组件只挂在批量渲染对象上，不按 primitive 粒度挂载。

第一阶段渲染适配层可以采用两种保守方式：

- 批量 mesh：把一个逻辑对象内的 primitive 烘成一个或少量 mesh。颜色可以优先用顶点色；如果标准材质路径限制较多，则临时按颜色或材质分组。
- 简化实例数据：如果实现成本可控，直接让逻辑对象生成 cube / sphere instance 数据，为后续实例池铺路。

这个阶段重点不是极限性能，而是验证：

- RON 蓝图能稳定读取。
- cube / sphere 显示正确。
- 坐标和尺寸符合世界观规则。
- 颜色和材质复用可行。
- 第一批家园蓝图有可读性。

短期优化：

- unit cube 和 unit sphere 作为基础模板或共享 mesh 源数据。
- 颜色相同的 primitive 复用材质。
- 单个 primitive 不拥有 ECS Transform、Visibility、Material 或逐帧逻辑。
- 所有 primitive 数据归属到对应逻辑 Entity 的 `FangyuanPrimitiveSet`。

### 第二阶段：静态实例池

当家园、宗门工程、天道装饰开始增长后，需要把第一阶段的批量 mesh 或标准渲染适配替换为静态实例池。

目标：

```text
大量静态 primitive
-> 按 cube / sphere 分组
-> 写入静态 instance buffer
-> 一批或少量 draw call 渲染
```

静态内容包括：

- 家园主体结构。
- 宗门工程。
- 方块碑林。
- 浮空灯。
- 区域装饰。
- 纪元档案展示物。
- 低频变化的天道自演装饰。

静态实例池特点：

- 构建后很少变化。
- 修改蓝图时重建局部或整组 instance 数据。
- primitive 不参与逐帧 ECS Transform 更新。
- 可按空间块、逻辑对象 root 或蓝图 id 管理。

### 第三阶段：动态 VFX 实例池

技能、护盾、投射物、命中反馈和拖尾需要动态播放。

动态 primitive 同样不成为玩法 Entity。从第一版 VFX 开始就应使用 VFX 实例池：

```text
一个技能释放事件
-> 一个 VFX runtime instance
-> 内部持有 recipe、start_tick、seed、caster、target
-> 每帧批量计算当前可见 primitive
-> 写入动态 instance buffer
```

动态实例池特点：

- 生命周期短。
- 数量波动大。
- 需要按距离和热点快速降级。
- 需要支持确定性回放。
- 需要和 authority tick 对齐。

### 第四阶段：统一 FangyuanMaterial

当颜色、发光、透明和材质 profile 增多后，需要统一方圆材质。

推荐实例数据：

```text
InstanceData:
  position: vec3
  scale: vec3
  color: vec4
  emissive: f32
  alpha: f32
  material_profile_id: u32
  flags: u32
```

如果后续确认所有 primitive 永远无旋转，可以不传完整 transform matrix，而是传 position + scale。这样实例数据更小，shader 中顶点位置也更简单：

```text
world_position = instance_position + local_vertex_position * instance_scale
```

材质 profile 可以表达：

- 普通实体。
- 金属。
- 石质。
- 玉石。
- 灵木。
- 火核。
- 水膜。
- 风灵。
- 天道碑。
- 纪元档案。

早期 profile 可以只是参数表，不必马上接 texture array。

### 第五阶段：texture atlas 或 texture array

当单纯颜色和 profile 参数不足以表达材质时，再考虑纹理。

候选：

- texture atlas：实现较直接，但 UV 边界和采样出血需要处理。
- texture array：更适合按材质层采样，但管线和资源管理更复杂。

在方圆灵构中，纹理不应成为主要识别来源。纹理适合做轻微质感：

- 石纹。
- 金属噪声。
- 火核脉动。
- 水膜细波。
- 风灵流线。
- 纪元档案符纹。

不建议允许玩家上传任意纹理作为早期能力。任意纹理会带来审核、版权、包体、缓存和性能问题。

## 各功能渲染路线

### 方圆洞府和家园

家园是方圆灵构最适合先落地的场景。

短期路线：

- 读取本地 RON 蓝图。
- 生成一个或少量 `HomeObjectEntity` / `FangyuanObjectEntity`。
- 蓝图 primitive 存入逻辑 Entity 的 `FangyuanPrimitiveSet`。
- 渲染适配层生成批量 mesh 或简化 instance 数据。
- 复用 mesh 和颜色材质。
- 控制 primitive 数量在 1000 以内。
- 保证坐标、尺寸、颜色合法。

中期路线：

- 家园蓝图进入统一 validator。
- 家园内容编译为 runtime primitive。
- 静态 primitive 进入静态实例池。
- 家园逻辑 Entity 只保留状态、交互、选择和 primitive set handle。
- 编辑或重新生成后局部重建实例数据。

长期路线：

- 家园支持跨纪元继承和重新落位。
- 大型家园按空间块或楼层拆分实例池。
- 远处家园显示低精度代理。
- 访问他人家园时按权限和距离流式加载。
- 家园访客、天道自演 NPC 和互动装饰使用独立生命周期。

渲染重点：

- 主结构清楚。
- 方相承担承载和边界。
- 圆相承担炉心、灵核、阵眼和流动感。
- 不靠真实建筑模型堆细节，而靠方圆组合形成身份。

### 宗门工程和公共设施

宗门工程比个人家园更大，不能简单提高 primitive 上限。

推荐：

- 宗门工程按设施拆分：界碑、传送阵、法会场、公共须弥锚点、工坊、分舵。
- 每个设施有独立蓝图和预算。
- 同屏只加载附近设施完整表现。
- 远处显示轮廓或地标核心。
- 人群热点时优先保留可交互核心和导航识别。

宗门工程适合使用更稳定、更厚重的方相结构，圆相作为核心节点、能量连线和传送标记。

### 装备外观

装备是自由度中等的方圆灵构对象。

装备渲染需要解决两个问题：

1. 外观看起来可自定义。
2. 技能能稳定从装备上发出或附着。

推荐装备蓝图提供语义挂点：

- `grip`：握持点。
- `tip`：发射或指向点。
- `core`：圆相核心。
- `guard`：防护结构。
- `aura`：灵气附着区域。

装备本体仍由 cube / sphere 组合。挂点不是骨骼，而是从蓝图或编译结果中计算出的语义位置。

短期可以将装备作为角色附近的独立方圆结构显示。长期如果需要绑定角色动作，需要谨慎处理，因为当前基础规则不支持 primitive 旋转。

如果角色动画后续使用传统模型，装备方圆结构可以绑定到角色手部或背部 root，但装备内部 primitive 仍不旋转，只跟随装备 root 的整体位姿。这个方向需要后续单独评估，因为它会引入“装备 root 旋转”和“primitive 不旋转”的边界。

### 技能和程序 VFX

技能表现必须以模板为边界。

服务端识别技能模板：

- 投射物。
- 范围效果。
- 持续区域。
- 护盾。
- 治疗。
- 控制。
- 位移。
- 召唤物。
- 陷阱。

客户端视觉由 `SkillVisualBlueprint` 和程序动画 recipe 还原。

推荐技能表现拆成四层：

```text
规则层：真实范围、方向、命中区域
结构层：方块阵列、球形核心
能量层：小球、小方块、拖尾、闪烁、扩散、收束
残留层：命中后的淡出、碎散、余辉、地面短暂标记
```

基础动画能力：

- spawn。
- move。
- scale。
- fade。
- color_shift。
- emissive_pulse。
- trail。
- impact_expand。
- dissolve。
- attach_to_entity。

不作为基础能力：

- primitive 自由旋转。
- 骨骼动画。
- mesh deformation。
- 玩家上传 shader。
- 任意脚本。

无旋转限制下，斩击和扇形技能可以这样表达：

```text
不用旋转刀光
改用一串球体或方块沿弧线依次显化、膨胀、淡出
```

### NPC 和天道访客

方圆灵构世界中的 NPC 不必一开始走传统角色模型路线。

天道自演 NPC 可以先采用方圆抽象体：

- 方相表达身体、衣摆、护甲、身份结构。
- 圆相表达头部、灵核、眼目、法器核心。
- 发光和颜色表达四属性。
- 小幅移动、缩放、显隐和脉冲表达生命感。

这类 NPC 更适合纯表现层和弱互动层：

- 闲聊。
- 传闻。
- 纪元记录。
- 家园访客评论。
- 区域提示。

如果后续要引入高表现角色模型，应明确分层：

- 主角和关键角色可以使用传统角色渲染。
- 天道自演、大量访客、低成本 NPC 继续使用方圆抽象体。
- 大规模场景下优先保留方圆 NPC，因为它们更可审核、可降级、可批处理。

### 天道自演生成物

天道自演生成物必须从第一天就考虑生命周期和预算。

推荐状态：

- 灵感期：只有生成意图，不渲染。
- 显化期：蓝图审核通过后出现。
- 活跃期：玩家可查看、点赞、收藏、举报或轻互动。
- 衰退期：低因果值内容逐渐淡出或隐藏。
- 固化期：高因果值内容转为长期世界记忆。
- 归档期：保留记录，不一定继续渲染。

渲染层需要支持：

- 生成物显化。
- 发光脉冲。
- 因果值提升时增强表现。
- 衰退时淡出或降级。
- 固化后转入静态实例池。
- 举报或预算回收后清理。

天道自演内容不应默认永久存在。否则长期世界会被生成物堆满，渲染和内容管理都会失控。

### 世界事件和纪元异象

纪元更替、天门开启、火脉复苏、地脉稳固、风门大开、水泽归元等事件需要高识别度。

推荐做法：

- 大型异象由少数高语义 primitive 组成。
- 远处可见核心轮廓。
- 近处补充能量层和残留层。
- 事件边界和危险区域必须清楚。
- 大型异象不能使用无限粒子和无限透明叠加。

例如天门裂隙可以表达为：

- 方相阵基构成门框和界碑。
- 圆相核心沿门缝排列。
- 发光强度随纪元倒计时变化。
- 远处只显示门框轮廓和中心圆核。

### 须弥空间和背包表现

背包和须弥空间本质是 UI 与道具系统，但也可以有世界观渲染表达。

建议：

- 背包 UI 保持清晰、实用，不要过度装饰。
- 须弥空间打开时可以用方圆灵构做背景表现。
- 地属性容量可表现为厚重方相格架。
- 火属性可表现为炉心和转化圆核。
- 水属性可表现为柔和圆相保存层。
- 风属性可表现为远程调度节点和轻量悬浮方块。

这些表现不应影响道具操作效率。UI 可读性优先于视觉炫技。

### 灵名溯源和纪元档案

灵名溯源、因果名录和纪元档案主要是 UI 和信息展示，但也可以使用方圆视觉语言：

- 称号和纪元记录可显示为小型方圆徽记。
- 高关注角色可有天道记录标记。
- 同名风险提示必须清楚，不能只靠美术符号。
- 纪元档案可以在家园或宗门空间中显示为方相碑和圆相记忆核心。

渲染原则：

- 信息真实性优先。
- 装饰不能遮挡身份字段。
- 可交互资料卡不应过度动态化。

## LOD 和预算路线

### LOD 分层

推荐表现档位：

```text
L0 自身完整表现：100 到 300 primitives
L1 队友或目标重点表现：40 到 100 primitives
L2 普通附近表现：10 到 40 primitives
L3 远处表现：1 到 8 primitives
L4 极端热点表现：只显示规则轮廓或范围提示
```

家园和场景也应有类似档位：

```text
近处完整结构
中距离保留主体和核心
远处保留剪影、地标和发光核心
极远只保留地图标记或不渲染
```

### 预算类型

灵构额度应同时承担世界观预算和技术预算。

推荐预算项：

- primitive 数量。
- cube 数量。
- sphere 数量。
- 动态 primitive 数量。
- 透明 primitive 数量。
- 发光 primitive 数量。
- 发光强度总量。
- 生命周期长度。
- 同屏技能数量。
- 遮挡面积。
- 规则层可读性。
- 材质 profile 复杂度。
- 更新频率。
- 网络蓝图引用数量。

技术预算不应在 UI 中完全裸露为工程术语。可以包装为：

- 灵构额度。
- 天道算力。
- 显化稳定度。
- 灵压占用。
- 阵列负载。

但调试面板需要保留真实指标。

### 热点区域降级

当天道灵压升高时，渲染降级应按顺序执行：

1. 降低远处装饰层。
2. 降低透明和发光强度。
3. 降低技能拖尾和残留。
4. 降低非关键天道自演生成物表现。
5. 降低普通玩家装备外观细节。
6. 保留技能规则层、危险边界和交互核心。
7. 必要时隐藏非关键展示实体。

不能优先移除：

- 敌方技能真实范围。
- 重要交互提示。
- 传送门、出口、任务核心。
- 安全区边界。
- 同名身份和交易风险提示。

## 多人和网络同步

### AOI 是必须项

长期多人场景不能全量同步、全量渲染。

需要 AOI 和兴趣管理：

- 只同步附近玩家。
- 只同步相关技能事件。
- 远处玩家低频同步。
- 远处家园和宗门工程只同步代理信息。
- 天道自演生成物按区域和可见性加载。

客户端不应知道也不应渲染整个世界所有蓝图。

### 蓝图缓存和哈希

玩家自定义蓝图和天道生成蓝图需要缓存。

推荐：

- 服务端审核后生成 `visual_blueprint_id`。
- 蓝图内容带版本和 hash。
- 客户端本地缓存蓝图。
- 技能释放只同步蓝图 id。
- 客户端缺蓝图时先显示降级默认表现，再异步请求蓝图。

这样可以避免每次技能释放都传一大段视觉数据。

### authority 回放

当前项目已有 authority 帧同步回放输入的方向。方圆技能和 VFX 应尽量和 authority tick 对齐。

推荐同步字段：

```text
tick
caster_id
skill_template_id
visual_blueprint_id
position
direction 或 target
seed
```

客户端用 `tick` 和 `seed` 还原视觉生命周期。这样录制、回放、观战和多客户端一致性更容易控制。

## 审核和安全

### 蓝图审核

所有玩家生成和天道自演生成的内容都必须审核。

审核重点：

- 是否只使用 cube / sphere。
- 是否超出 primitive 数量。
- 是否超出总体积。
- 是否超出 bounds。
- 是否有地面以下主体结构。
- 是否有过多透明叠加。
- 是否有过强发光。
- 是否遮挡关键交互。
- 是否隐藏技能真实范围。
- 是否伪装成系统提示、排行榜身份或交易确认。
- 是否包含违规文本或现实图标。
- 是否超出角色、职业、四属性和世界层级的灵构额度。

### 审核失败不应只返回失败

更好的体验是返回替代方案：

- 缩小整体尺寸。
- 减少装饰层。
- 降低发光。
- 移除透明。
- 固定规则边界。
- 替换未解锁材质。
- 降级为合法模板表现。

世界观表达：

```text
天道算力不足，已为你推演稳定版本
```

### 玩家自定义 shader 不进入早期能力

不建议早期允许玩家上传 shader、脚本或任意材质程序。

原因：

- 安全风险高。
- 审核难度高。
- 性能不可控。
- 移动端兼容风险高。
- 会破坏统一方圆风格。

玩家自定义应先限制在：

- primitive 组合。
- 颜色。
- 发光强度。
- 材质 profile。
- 动画 recipe 参数。
- 装饰层密度。

## 调试和验收工具

长期必须建设调试工具，否则预算和降级会很难维护。

### 灵构试炼场

灵构试炼场不仅是玩法，也应是渲染验收工具。

试炼场需要展示：

- primitive 总数。
- cube / sphere 数量。
- 动态 primitive 数量。
- 透明数量。
- 发光预算。
- 材质 profile 数量。
- 当前 LOD 档位。
- 热点降级结果。
- 规则层是否清楚。
- 审核状态。
- 降级建议。

技能试炼场需要验证：

- 单技能表现是否清楚。
- 多技能叠加是否遮挡。
- 自身、队友、敌人、远处对象的 LOD 是否正确。
- authority 回放是否稳定。
- 同一个 seed 和 start tick 是否还原相同表现。

### 开发调试面板

开发面板至少应显示：

- 当前可见 FangyuanPrimitive 数。
- 静态实例数量。
- 动态实例数量。
- cube draw batch 数。
- sphere draw batch 数。
- 材质 batch 数。
- 透明 batch 数。
- GPU buffer 重建次数。
- 蓝图加载数量。
- 蓝图审核失败原因。
- LOD 降级原因。
- 热点灵压等级。

这些指标比单纯 FPS 更有用，因为它们能定位到底是 primitive 过多、透明过多、材质切换过多，还是动态 buffer 更新过多。

## 阶段开发路线

### 阶段 1：本地家园蓝图预览打稳

目标：

- 稳定读取 `project/assets/fangyuan/home_preview.ron`。
- 在当前 Bevy 场景中显示静态 cube / sphere 家园预览。
- 验证方圆灵构世界观是否成立。

技术做法：

- 生成 `HomeObjectEntity` 或 `FangyuanObjectEntity` 逻辑对象。
- 蓝图 primitives 存入 `FangyuanPrimitiveSet`。
- unit cube 和 unit sphere 作为基础模板或共享 mesh 源数据。
- 按颜色缓存复用材质。
- 渲染适配层按 cube / sphere / material 生成批量 mesh 或简化 render batch。
- 每个 primitive 不挂 ECS Transform、Visibility、Material 或逐帧逻辑。
- 非法 primitive 跳过并记录日志。

验收标准：

- RON 语法错误不会导致游戏崩溃。
- `primitives.len()` 不超过 1000。
- 所有 kind 只允许 `cube` 和 `sphere`。
- 不生成地面以下主体结构。
- 默认蓝图能清楚表达家园主题。
- `cargo fmt` 和 `cargo check` 通过。

主要风险：

- 蓝图生成随机噪音太多，缺少造型可读性。
- 颜色材质过多导致 batch 分裂，但第一阶段可以接受。
- 过早做复杂渲染会拖慢世界观验证。

### 阶段 2：统一方圆数据模型

目标：

- 家园、装备、技能、NPC、天道自演都能共享统一 primitive 结构。

技术做法：

- 增加 `FangyuanPrimitive` 或同等结构。
- 区分蓝图 primitive 和 runtime primitive。
- 增加 role、material_profile、emissive、alpha 等扩展字段的设计位置。
- 当前 RON v1 可先只填 kind、position、size、color。

验收标准：

- 当前家园蓝图可以编译为统一 runtime primitive。
- 后续装备和技能不用重新定义一套几何基础。
- 文档和代码命名能区分 blueprint、runtime、render instance。

主要风险：

- 过度抽象导致第一阶段开发变慢。
- 字段一次性设计过多但没人使用。

控制方式：

- 结构允许扩展，但实现只落地当前必要字段。

### 阶段 3：蓝图审核和预算系统

目标：

- 把世界观限制和技术限制统一成可执行审核。

技术做法：

- 实现 primitive 数量检查。
- 实现 bounds 检查。
- 实现 size 和 color 范围检查。
- 实现 alpha、emissive、material profile 的预留检查。
- 审核结果返回错误列表和可降级建议。

验收标准：

- 非法蓝图不会生成非法内容。
- 审核失败能说明原因。
- 未来可以按角色、四属性、职业和世界层级接入灵构额度。

主要风险：

- 审核只做技术校验，无法约束战斗可读性。

控制方式：

- 从阶段 3 开始引入 `role`，为后续规则层和个性层分离做准备。

### 阶段 4：静态实例化渲染原型

目标：

- 把大量静态方圆 primitive 从第一阶段批量 mesh 或标准渲染适配迁移到静态实例化渲染原型。

技术做法：

- 保留家园物件、宗门设施、天道装饰等逻辑 Entity。
- primitive 数据写入连续数组。
- 按 cube / sphere 分组。
- 构造静态 instance buffer。
- 渲染层使用共享 mesh 和实例数据。

验收标准：

- 数万静态 primitive 不引入对应数量的 Entity。
- 渲染提交成本主要随批次、材质和可见对象增长，而不是随玩法 Entity 数量线性膨胀。
- 不同颜色和大小可以在同一渲染路径中表达。
- 蓝图变更后可以重建实例数据。

主要风险：

- Bevy 自定义渲染管线接入复杂。
- 调试难度上升。

控制方式：

- 先保留第一阶段批量 mesh 或标准渲染适配作为 fallback。
- 只对静态家园或测试场景启用实例化路径。

### 阶段 5：统一方圆材质

目标：

- 支持多颜色、多发光、多基础材质感，同时尽量不打碎渲染批次。

技术做法：

- 定义 `FangyuanMaterial`。
- 实例数据携带 color、alpha、emissive、material_profile_id。
- material profile 初期用参数表表达。
- 默认不透明，透明走单独预算和路径。

验收标准：

- 同屏大量不同颜色 primitive 不需要创建大量材质资源。
- 发光强度可以按实例变化。
- 材质 profile 可以表达基础视觉差异。

主要风险：

- 半透明和不透明混在一起导致排序和性能问题。

控制方式：

- 不透明和透明分通道。
- 透明 primitive 数量设硬预算。

### 阶段 6：程序 VFX 播放器

目标：

- 支持技能和短生命周期表现，从第一版开始就不创建动态 primitive Entity。

技术做法：

- 增加 VFX recipe。
- 支持 spawn、move、scale、fade、color_shift、emissive_pulse、trail、impact_expand。
- 一个 VFX 实例内部批量计算 primitive 状态。
- 动态结果写入动态 instance buffer。

验收标准：

- 投射物、范围、护盾、命中扩散可以用方圆 primitive 表达。
- 同一个 start tick 和 seed 可以稳定回放。
- 单技能可以在试炼场中显示预算。

主要风险：

- 表现容易像工程调试效果。

控制方式：

- 用颜色曲线、发光、拖尾、命中反馈和残留层增加表现厚度。

### 阶段 7：技能规则层和个性层

目标：

- 玩家自定义技能视觉不破坏战斗可读性。

技术做法：

- 技能模板定义规则层。
- 视觉蓝图定义个性层。
- 规则层强制显示真实范围、方向和危险边界。
- 个性层接受预算和 LOD 控制。

验收标准：

- 同一技能模板在不同玩家皮肤下仍能识别。
- 视觉范围不误导真实作用范围。
- 热点降级时仍保留规则层。

主要风险：

- 玩家追求个性导致遮挡和误导。

控制方式：

- 审核遮挡面积、发光强度、颜色冲突和规则层可见性。

### 阶段 8：装备、NPC 和天道自演接入

目标：

- 让非家园内容也复用方圆渲染底座。

技术做法：

- 装备蓝图编译为 runtime primitive。
- 装备提供 grip、tip、core、guard、aura 语义挂点。
- 天道自演生成物使用生命周期状态机。
- 方圆 NPC 使用低成本抽象结构。

验收标准：

- 技能可以引用装备挂点。
- 天道生成物可以显化、衰退、固化和回收。
- NPC 和装饰可以按预算降级。

主要风险：

- 装备和 NPC 需求逼迫引入传统模型复杂度。

控制方式：

- 关键角色模型和方圆抽象 NPC 分层处理，不把所有对象都推向同一美术成本。

### 阶段 9：LOD、AOI 和热点降级

目标：

- 支撑多人同屏和天道灵压。

技术做法：

- 实现 L0 到 L4 表现档位。
- 接入距离、屏幕占比、重要性、队伍关系和灵压等级。
- AOI 决定加载和同步范围。
- 热点优先降低装饰层、透明、发光和拖尾。

验收标准：

- 多技能叠加时帧率可控。
- 远处玩家不会完整播放高成本技能。
- 热点区域降级后仍保留规则可读性。

主要风险：

- 玩家感觉系统粗暴削弱表现。

控制方式：

- 用天道灵压做预警和包装。
- 保留自身完整表现或较高优先级表现。

### 阶段 10：蓝图缓存、流式加载和纪元继承

目标：

- 支撑长期 MMO 内容积累。

技术做法：

- 蓝图审核后生成 id、version、hash。
- 客户端缓存常用蓝图。
- 缺失蓝图时显示降级默认表现。
- 家园、装备和技能蓝图可随纪元继承，但进入新世界重新校准预算。
- 天道自演固化内容进入区域或纪元档案。

验收标准：

- 不需要每次同步完整视觉蓝图。
- 纪元迁移不会清空玩家视觉资产。
- 新世界能按当地规则降级旧蓝图。

主要风险：

- 历史蓝图版本过多，兼容复杂。

控制方式：

- 蓝图版本化。
- 编译器提供旧版本升级或兼容路径。

### 阶段 11：试炼场和完整调试工具

目标：

- 让玩家理解预算，让开发者定位性能。

技术做法：

- 灵构试炼场展示审核和预算结果。
- 开发调试面板展示实例数量、batch、buffer 更新、LOD 和灵压。
- 压力测试场景模拟 100、300、1000 人技能释放。

验收标准：

- 玩家能看到蓝图为什么被降级。
- 开发者能定位性能瓶颈。
- authority 回放下视觉一致性可验证。

主要风险：

- 没有工具时，后续每次优化都靠猜。

控制方式：

- 试炼场和调试面板不应拖到最后才做，至少从阶段 6 开始逐步建设。

## 推荐优先级

如果只看近期开发，建议优先顺序是：

1. 家园蓝图预览稳定。
2. 统一方圆 primitive 数据模型。
3. 蓝图审核和预算。
4. 静态实例化原型。
5. 程序 VFX 播放器。
6. 技能规则层和个性层。
7. LOD 和热点降级。
8. 装备、NPC、天道自演全面接入。
9. 蓝图缓存、流式加载和纪元继承。
10. 大规模压力测试和长期调优。

原因：

- 家园是最安全的创作对象，可以先验证世界观。
- 统一数据模型越早做，后续返工越少。
- 审核和预算是 AI 生成内容的底线。
- 静态实例化先于动态 VFX，风险更低。
- 技能和多人热点最复杂，应等底座稳定后推进。

## 关键风险

### 风险一：过早追求自定义渲染

自定义渲染管线会提高性能上限，但也会带来复杂度。

控制方式：

- 第一阶段先用 Bevy 标准渲染验证玩法。
- 实例化渲染先做独立原型。
- 保留第一阶段批量 mesh 或标准渲染适配 fallback，直到实例化路径稳定。

### 风险二：primitive 泄漏为玩法 Entity

正式架构如果出现一个 primitive 对应一个玩法 Entity，就会破坏从第一阶段确立的逻辑对象边界，并且很快变成 ECS、Extract 和可见性计算的性能债。

控制方式：

- 从第一阶段开始禁止把 primitive 作为玩法 Entity。
- 家园、技能、装备、NPC、天道生成物都只能持有 primitive 数据集或其 handle。
- 如果为了调试临时生成 primitive 可视对象，必须隔离在 debug-only 路径，不纳入正式架构。
- code review 时重点检查是否新增了 primitive 粒度的 Transform、Visibility、Material 和逐帧系统。

### 风险三：材质自由导致批次爆炸

玩家想要丰富颜色和材质，但底层不能为每种视觉差异创建独立材质。

控制方式：

- 实例数据携带颜色和 profile。
- 材质 profile 数量受控。
- 纹理能力后置。

### 风险四：透明和发光失控

透明和发光最容易造成视觉遮挡和性能问题。

控制方式：

- 默认不透明。
- 透明单独预算。
- 发光强度总量预算。
- 热点优先压制透明和发光。

### 风险五：技能自定义破坏可读性

玩家可能把危险技能做成不明显，或把无害装饰做成强危险提示。

控制方式：

- 系统固定规则层。
- 玩家只控制个性层。
- 审核真实范围和视觉范围一致性。
- 试炼场检查多技能叠加。

### 风险六：天道自演内容永久堆积

如果生成物没有生命周期，世界会越来越重。

控制方式：

- 默认临时。
- 因果值决定固化。
- 衰退和回收释放预算。
- 固化内容也需要按区域和距离加载。

## 决策口径

方圆灵构长期渲染方向是：

```text
受控几何实例动画
而不是任意模型和任意特效资源系统
```

家园、装备、技能、NPC、场景道具和天道自演都应尽量收敛到：

```text
cube / sphere
position / scale
color / alpha / emissive
material profile
role
budget
LOD
```

1000 人热点场景的可行前提是：

```text
服务端只模拟规则模板
网络只同步语义事件
客户端只渲染 AOI 内且预算内的表现
自定义只影响可降级视觉层
规则层永远清楚
```

这条路线牺牲了传统美术资产的自由度，但换来：

- 世界观统一。
- AI 生成可审核。
- 玩家创作可继承。
- 渲染底层可扩展。
- 多人热点可降级。
- 移动端有优化空间。
