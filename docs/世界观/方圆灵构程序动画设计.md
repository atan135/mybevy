# 方圆灵构程序动画设计

## 文档定位

本文记录方圆灵构体系下，装备和技能表现以程序动画为主时的技术与设计口径。

本文不是具体实现方案，不代表动画系统、技能系统、蓝图审核、渲染管线或服务端同步已经实现。落地时应继续拆分为动画蓝图格式、技能模板、客户端渲染、服务端同步、性能预算和试炼场验收规则等具体设计。

本文与这些文档相关：

- [方圆灵构](./方圆灵构.md)：定义方相、圆相、灵构蓝图、灵构额度和 AI 生成边界。
- [方圆灵构蓝图规则](./方圆灵构蓝图规则.md)：定义第一阶段家园静态蓝图的本地 RON 格式。
- [玩法体系设想](./玩法体系设想.md)：定义灵构工坊、灵构试炼场、装备和技能自定义的玩法方向。
- [天道灵压](./天道灵压.md)：定义区域压力、热点分流和技术限压的世界观包装。

## 核心判断

方圆灵构下的动画不应优先采用传统美术动画管线，即不应把核心能力建立在大量预制 clip、骨骼动画或每个技能独立特效资源上。

更适合的方向是：

```text
技能或装备规则模板
-> 方圆灵构蓝图
-> 程序动画配方
-> 审核与预算
-> 客户端运行时表现
-> authority 事件回放
```

玩家可以自定义技能和装备的视觉表达，但规则强度、命中范围、控制时间、冷却、消耗和服务端判定必须由系统模板决定。

核心原则：

```text
系统模板决定规则强度
玩家蓝图决定视觉表达
程序动画决定运行时表现
审核预算决定边界
```

## 动画约束

当前推荐将方圆灵构程序动画限制在较小但稳定的表现空间内。

基础 primitive：

- `cube`：方相，表达结构、边界、阵列、护盾、规则和承载。
- `sphere`：圆相，表达灵核、能量、爆发、流动和感知。

基础动画能力：

- 创建。
- 消失。
- 移动。
- 缩放。
- 颜色变化。
- 透明度变化。
- 发光强度变化。

暂不作为基础能力：

- primitive 旋转。
- 骨骼动画。
- mesh deformation。
- 任意脚本。
- 任意 shader 上传。
- 每个玩家自定义渲染程序。

不允许旋转会削弱斩击、扇形扫击、斜向光束和复杂符文表现，但可以换成方圆阵列语言：

```text
不用旋转刀光
改用一串球体或方块沿弧线依次出现、缩放、淡出
```

这种表达更抽象，但更贴合方圆二相，也更容易审核、同步和降级。

## 动画表现配方

技能表现不应只是一批静态 primitive，而应拆成可组合的程序动画配方。

推荐基础通道：

```text
position = f(start_pos, target_pos, t, seed)
scale = g(t)
color = h(t)
alpha = a(t)
emissive = e(t)
```

推荐基础动作：

- `spawn`：显化。
- `move`：沿直线、弧线、折线或预定义轨迹移动。
- `scale`：膨胀、收缩、脉冲。
- `fade`：淡入、淡出。
- `color_shift`：颜色过渡。
- `pulse`：周期性缩放或亮度变化。
- `trail`：拖尾残影。
- `impact_expand`：命中扩散。
- `dissolve`：溶解消散。
- `attach_to_entity`：绑定到角色、装备或场景锚点。

示例：投射物技能表现。

```text
中心层：一个高亮圆相球向目标方向移动
结构层：若干方相块沿路径依次显化
拖尾层：小圆相逐渐缩小并淡出
命中层：圆相短暂膨胀，方相向外扩散后消失
```

示例：护盾技能表现。

```text
边界层：方相组成保护边界
节点层：圆相作为能量核心
反馈层：受击位置局部放大、变色、回弹
消散层：方相下沉或缩小，圆相亮度归零
```

示例：范围技能表现。

```text
预警层：地面方相网格逐渐显化
爆发层：中心圆相膨胀
持续层：小圆相在范围内脉冲
结束层：方相下沉或淡出
```

## 技能模板与视觉蓝图

技能自定义必须以模板为边界。

服务端和规则层只承认有限技能模板：

- 投射物。
- 范围效果。
- 持续区域。
- 护盾。
- 治疗。
- 控制。
- 位移。
- 召唤物。
- 陷阱。

玩家可以自定义：

- 方相和圆相比例。
- 颜色倾向。
- 拖尾形态。
- 显化节奏。
- 命中扩散方式。
- 视觉层级数量。
- 装饰 primitive 的分布。

玩家不能自定义：

- 真实伤害。
- 真实控制时间。
- 真实命中范围。
- 服务端碰撞体数量。
- 技能冷却和消耗。
- 不可见或误导性的真实作用区域。
- 无限生命周期或无限 primitive 生成。

因此，技能模板和视觉蓝图应保持分离。

```text
SkillTemplate:
  rule_shape
  damage_budget
  control_budget
  cooldown
  cost
  authority_behavior

SkillVisualBlueprint:
  primitive_count
  primitive_roles
  motion_channels
  scale_channels
  color_channels
  lifetime
  lod_profiles
```

## 装备挂点

每个玩家装备都可能不同，因此技能动画不应绑定某个固定模型或固定骨骼。

装备蓝图需要提供稳定挂点。

推荐挂点：

- `grip`：握持点。
- `tip`：发射、挥击或指向点。
- `core`：装备圆相核心。
- `guard`：防护或结构点。
- `aura`：光效或灵气附着区域。

技能视觉只引用挂点语义，不引用具体模型结构：

```text
从 tip 发射圆相核心
围绕 core 显化三个球体
在 aura 周围生成方相护框
```

这样装备外观可以高度自定义，技能表现仍然可以稳定连接。

## 可读层与个性层

自定义技能必须分成可读层和个性层。

可读层由系统控制，必须清楚表达真实规则：

- 投射物必须有清晰核心、方向和速度。
- 范围技能必须显示真实作用半径或作用边界。
- 护盾必须显示保护边界。
- 陷阱必须显示触发区域。
- 位移技能必须显示方向、起点或落点。

个性层由玩家蓝图控制：

- 颜色。
- 拖尾。
- 方相与圆相组合。
- 显化节奏。
- 命中扩散。
- 残留余辉。
- 装饰密度。

简化口径：

```text
规则轮廓系统定
视觉皮肤玩家定
```

## 服务端性能边界

服务端不应模拟每个视觉 primitive。

服务端只处理规则模板和少量判定体：

```text
caster_id
skill_template_id
start_tick
position
direction 或 target
seed
visual_blueprint_id
```

真实判定使用固定低复杂度形状：

- 投射物：1 个碰撞球、胶囊或简化盒。
- 范围技能：1 个圆形、球形、盒形或扇形区域。
- 护盾：1 个半径、包围盒或角色附属判定。
- 陷阱：1 个触发区域。

服务端复杂度应接近：

```text
O(玩家移动 + 活跃技能模板 * 空间查询候选数)
```

必须避免：

```text
O(活跃技能 * 场景玩家数)
O(活跃技能 * 自定义视觉 primitive 数 * 玩家数)
O(视觉 primitive 每帧同步)
```

1000 人同场景时，服务端可以处理的是规则事件和模板判定，不是完整视觉动画。

## 网络同步边界

1000 人场景不能做全量互相同步。

必须采用：

- AOI。
- 兴趣管理。
- 远处对象低频同步。
- 技能释放事件批处理。
- 视觉蓝图只同步 id。
- 蓝图内容提前审核、缓存、哈希化。

技能释放事件推荐同步：

```text
tick
caster_id
skill_template_id
visual_blueprint_id
position
direction 或 target
seed
```

客户端根据 `start_tick`、`seed` 和 `visual_blueprint_id` 本地还原表现。

不要同步：

- 每个 primitive 的每帧位置。
- 每个 primitive 的每帧缩放。
- 每个 primitive 的生命周期状态。
- 客户端视觉播放器内部状态。

## 客户端渲染边界

客户端不应把每个 primitive 都建成完整 Bevy Entity 并逐帧跑组件系统。

危险形态：

```text
1000 个玩家
每人 1 个活跃技能
每个技能 100 个 primitive
= 100000 个动态 primitive Entity
```

更适合的运行时结构：

```text
一个技能 VFX = 一个运行时实例
实例内部持有 primitive 数据数组
批量计算位置、缩放、颜色和透明度
按 cube / sphere 分批写入渲染实例数据
```

推荐渲染策略：

- cube 共享 unit cube mesh。
- sphere 共享 unit sphere mesh。
- 同材质和同 mesh 尽量批处理或 instancing。
- 动画数据用连续数组管理。
- 颜色、透明度、发光强度尽量批量更新。
- 能放到 shader 的时间函数尽量放到 shader。
- 技能生命周期用 `start_tick` 或 `start_time` 推导。

客户端复杂度应从：

```text
O(可见特效 * primitive * ECS 逐实体更新)
```

降到：

```text
O(可见特效实例 + GPU 批量绘制)
```

## 表现厚度

仅有创建、移动、缩放和消失时，视觉容易变成工程调试效果。

表现厚度不应通过增加 primitive 类型解决，而应通过这些方式解决：

- 多阶段生命周期。
- 颜色曲线。
- 透明度曲线。
- 发光强度。
- emissive 材质。
- bloom 泛光。
- 边缘光。
- 噪声或溶解材质。
- 拖尾残影。
- 命中反馈。
- 屏幕轻微震动或闪光。
- 地面短暂范围标记。

技能表现建议拆成四层：

```text
规则层：真实范围、方向、命中区域，必须清楚
结构层：方块阵列、球形核心，体现技能身份
能量层：小球、小方块、拖尾、闪烁、扩散、收束
残留层：命中后的淡出、碎散、余辉、地面短暂标记
```

核心口径：

```text
形状保持简单
材质和时间节奏做丰富
```

## 表现预算

灵构额度应同时承担世界观预算和技术预算。

推荐把技能表现分为多档：

```text
自身完整表现：100 到 300 primitives
队友或目标重点表现：40 到 100 primitives
普通附近表现：10 到 40 primitives
远处表现：1 到 8 primitives
极端拥挤表现：只显示规则轮廓或范围提示
```

实际预算应按平台、场景和战斗密度动态调整。

移动端应更保守：

- 同屏高表现技能数量有限。
- 多数无关技能使用低表现。
- 远处玩家只保留规则可读层。
- 热点区域触发特效降级。

天道灵压可以承担热点降级和分流的世界观包装：

```text
区域灵压升高
-> 非关键特效降级
-> 远处灵构显化变弱
-> 技能装饰层被天道压制
-> 开启分流或迁移引导
```

## 审核规则

技能视觉蓝图必须可审核。

审核重点：

- primitive 数量。
- 生命周期长度。
- 同屏特效预算。
- 发光强度。
- 透明叠加层数。
- 遮挡面积。
- 颜色是否干扰敌我和危险提示。
- 真实作用范围是否清楚。
- 是否试图隐藏、误导或扩大规则层。
- 是否生成过多持续残留。
- 是否超出当前角色、职业、四属性和世界层级的灵构额度。

审核失败时应提供替代方案：

- 降低 primitive 数量。
- 缩短生命周期。
- 降低发光强度。
- 替换透明叠加。
- 固定规则轮廓。
- 降级为合法模板表现。

## 试炼场验收

灵构试炼场应成为自定义技能表现的验收入口。

试炼场至少需要检查：

- 单技能表现是否清楚。
- 多技能叠加是否遮挡。
- 自身、队友、敌人和远处对象的 LOD 是否正确。
- 低端设备或移动端预算下是否可接受。
- 技能真实范围和视觉范围是否一致。
- authority 回放时表现是否稳定。
- 同一个 `seed` 和 `start_tick` 是否能还原相同表现。

试炼场不应只展示最终效果，还应展示预算结果：

```text
primitive 使用量
同屏预算占用
发光预算占用
遮挡风险
LOD 降级结果
审核状态
```

## 推荐架构

推荐拆分为这些层。

```text
framework/animation/
  procedural_player
  animation_recipe
  curves
  vfx_instance_buffer
  lod
  budget
  debug

framework/fangyuan/
  blueprint
  validator
  compiler
  runtime_primitive

game/skills/
  skill_template
  skill_visual_mapping
  skill_authority_event
  trial_field

game/equipment/
  equipment_blueprint
  equipment_socket
  equipment_visual_runtime
```

分层职责：

- `fangyuan` 负责蓝图格式、审核和编译。
- `animation` 负责程序动画、曲线、实例数据和渲染提交。
- `skills` 负责规则模板、技能释放、authority 同步和表现映射。
- `equipment` 负责装备外观、挂点和技能附着。

## 阶段建议

第一阶段：静态与低频动态验证。

- 复用方圆灵构蓝图规则。
- 支持 cube / sphere。
- 支持创建、消失、移动、缩放和颜色变化。
- 不支持旋转。
- 不支持服务端判定接入。
- 支持单技能预览。

第二阶段：技能模板与表现映射。

- 增加投射物、范围、护盾等基础模板。
- 增加 `SkillVisualBlueprint`。
- 增加规则层和个性层。
- 增加本地试炼场。

第三阶段：性能预算与 LOD。

- 增加 primitive 预算。
- 增加自身、近处、远处和拥挤区域 LOD。
- 增加渲染批处理或 instancing。
- 增加预算调试面板。

第四阶段：authority 回放。

- 技能释放事件只同步模板、蓝图 id、tick、方向和 seed。
- 客户端按起始 tick 还原表现。
- 对比多客户端回放一致性。

第五阶段：热点场景验证。

- 构造 100、300、1000 人压力测试。
- 分别验证服务端事件吞吐、客户端渲染帧率、网络下行和 LOD 降级。
- 将天道灵压接入热点提示、特效降级和分流规则。

## 风险

主要风险：

- 1000 人场景如果全量同步，网络先成为瓶颈。
- 每个 primitive 一个 Entity 会带来明显 ECS 和调度压力。
- 大量透明叠加和动态光源会比 primitive 数量更早击穿客户端性能。
- 玩家自定义如果不分可读层和个性层，会破坏战斗可读性。
- 如果视觉 primitive 参与服务端判定，复杂度会迅速失控。
- 如果没有试炼场和预算反馈，玩家无法理解为什么蓝图被降级。

## 决策口径

方圆灵构程序动画的长期方向是：

```text
受控几何实例动画
而不是任意动画资源系统
```

1000 人场景的可行前提是：

```text
服务端只模拟规则模板
网络只同步语义事件
客户端只渲染可见且预算内的表现
自定义只影响可降级视觉层
```

球体、正方体、移动、缩放、颜色和发光足以形成清晰风格，但必须依赖材质、节奏、拖尾、命中反馈和 LOD 才能避免表现单薄。
