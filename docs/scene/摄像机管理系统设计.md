# 摄像机管理系统设计

本文记录 MyBevy 第一阶段摄像机管理系统的设计、实现范围和验收口径。本文是开发前设计文档，不表示所有能力已经落地；实现完成后需要同步更新“当前状态”和相关框架说明。

## 1. 目标

第一阶段目标是从文档到实现完成一套适配当前项目的摄像机管理能力，并应用到现有游戏玩法中。至少覆盖：

- 固定视角。
- 跟随视角。
- 可配置摄像机动画或过渡。
- 当前玩法接入与可验证验收。
- 镜头特效与渲染特效的职责边界。

当前项目已有 `project/src/framework/scene/camera.rs`，可以根据 scene manifest 生成基础 2D/3D 场景相机。第一阶段应在现有 `SceneCameraRig`、`SceneOwned(session_id)`、scene manifest 和全局 UI 摄像机层级上扩展，而不是另起一套不受场景生命周期管理的全局相机。

## 2. 约束

- 使用 Rust stable 和 `bevy = "0.18.1"`。
- 所有 Rust/Bevy 命令默认在 `project/` 目录执行。
- 框架层能力放在 `project/src/framework/scene/`。
- 游戏层接入放在 `project/src/game/scenes/` 或 `project/src/game/features/`。
- HUD 仍归 UI 框架管理，不挂到 `SceneRoot`。
- 场景相机顺序必须低于全局 UI 相机，避免遮挡 HUD。
- 随场景销毁的相机、目标标记和运行时状态必须能通过 `SceneOwned(session_id)` 或对应资源清理。
- 不改变 Robot Sync 的 authority/replay/network/MyServer 语义。
- Robot Sync 跟随相机必须保留远端玩家可见性或提供可回退的总览视角，避免联调时只看见本机。

## 3. 职责边界

### 3.1 Framework Layer

`project/src/framework/scene/` 负责通用摄像机能力：

- 解析 manifest 中的相机模式、投影、初始 Transform、目标和动画配置。
- 生成或复用场景相机实体。
- 维护 `SceneCameraRig` 运行状态。
- 每帧应用固定、跟随和动画过渡。
- 提供通用相机目标标记、查询和命令接口。
- 按 `SceneOwned(session_id)` 清理随场景生命周期创建的实体。
- 保持相机更新与 UI 相机、Loading、scene lifecycle 的边界清晰。

Framework layer 不负责：

- 判断某个玩法应该跟随哪个角色。
- 解释具体角色、怪物、战斗、技能和剧情语义。
- 直接读写 authority replay 状态。
- 实现后处理渲染管线。

### 3.2 Game Layer

`project/src/game/` 负责玩法相关策略：

- 在本地玩家、Boss、剧情点、交互目标等实体上挂相机目标标记。
- 根据玩法模式发起相机切换命令。
- 为具体场景选择默认相机配置。
- 定义 Robot Sync、样板场景、后续战斗或家园场景的镜头策略。
- 在 HUD 或调试配置中提供必要的切换入口。

### 3.3 Render/VFX Layer

渲染层负责后处理和屏幕空间效果，例如 bloom、vignette、color grading、motion blur、depth of field、chromatic aberration、屏幕空间扭曲、材质和粒子特效。

摄像机系统可以发出“镜头意图”或维护 transform/projection 类效果，但不应在第一阶段承载完整后处理实现。

## 4. 镜头特效归属

镜头特效需要纳入摄像机系统设计，但第一阶段只实现 transform/projection 类基础效果。

归摄像机系统：

- 镜头震动。
- FOV kick 或短暂 zoom pulse。
- camera roll。
- 固定视角与跟随视角切换。
- 过场镜头轨迹。
- hit stop 期间的镜头冻结或缓动。
- 目标切换、拉近、拉远。

归渲染或 VFX 系统：

- bloom。
- vignette。
- motion blur。
- depth of field。
- chromatic aberration。
- color grading。
- 屏幕空间扭曲。
- 材质、粒子、拖尾、技能光效。

协作方式：

- 摄像机系统发出 `CameraImpact`、`CameraFocusChanged`、`CameraEffectIntent` 这类意图事件。
- 渲染系统按设备档位和画质配置决定是否开启后处理。
- 第一阶段只预留事件和文档边界，不实现完整后处理栈。

## 5. 第一阶段能力

### 5.1 固定视角

固定视角是当前已有能力的增强和规范化：

- manifest 声明 `mode: "fixed3d"` 或 2D 固定相机。
- 配置 `position`、`rotation`、`projection`。
- 可选配置 `target` anchor，用于校验或辅助 `looking_at`。
- 进入场景时创建相机，并挂 `SceneCameraRig`、`SceneOwned(session_id)` 和 `Name`。
- 退出场景时无残留。

适用场景：

- Robot Sync 总览。
- 家园或样板场景预览。
- 需要稳定构图的调试和验收。

### 5.2 跟随视角

跟随视角应支持动态实体和静态 anchor 两类目标。

目标来源建议：

- `Anchor(SceneAnchorId)`：跟随 manifest 中的稳定 anchor。
- `Entity(Entity)`：跟随指定实体，通常由命令绑定。
- `TargetTag(String)`：查找当前 session 下挂有指定相机目标标记的实体。
- `LocalPlayer`：游戏层把本地玩家 visual 标记为本地跟随目标，framework 不直接理解玩家协议。

跟随参数建议：

- `offset`：相机相对目标的位置偏移。
- `look_at_offset`：看向点相对目标的位置偏移。
- `position_lerp`：位置平滑系数。
- `rotation_lerp`：旋转平滑系数。
- `min_distance` / `max_distance`：可选距离限制。
- `bounds`：可选相机边界。
- `keep_targets_visible_radius`：可选保底可见半径，用于 Robot Sync 等多人联调场景。

Robot Sync 中的默认跟随策略不应把镜头贴得太近。推荐保留高俯视 offset，并提供 overview fallback。

### 5.3 摄像机动画和过渡

第一阶段优先做轻量配置式 tween，不做完整 timeline 编辑器。

基础动画能力：

- 进入场景时从初始 Transform 平滑到目标 Transform。
- 固定视角切换到跟随视角时平滑过渡。
- 跟随视角切回总览时平滑过渡。
- 可选插值 FOV。
- 支持 duration 和 easing。

建议支持的 easing：

- `linear`
- `smooth_step`
- `ease_in_out`

暂不做：

- 多轨道电影 timeline。
- 复杂曲线编辑。
- 摄像机剪辑资源格式。
- 与剧情系统强绑定的 cutscene 编排。

## 6. 数据结构草案

以下是设计草案，具体命名可在实现时按代码风格调整。

```rust
pub enum SceneCameraMode {
    UiOnly2d,
    Gameplay2d,
    Gameplay3d,
    Fixed3d,
    FollowTarget,
    DebugFree,
}

pub struct SceneCameraConfig {
    pub mode: SceneCameraMode,
    pub transform: Transform,
    pub projection: SceneCameraProjection,
    pub target: Option<SceneAnchorId>,
    pub follow: Option<SceneCameraFollowConfig>,
    pub animation: Option<SceneCameraAnimationConfig>,
}

pub struct SceneCameraFollowConfig {
    pub target: SceneCameraTargetRef,
    pub offset: Vec3,
    pub look_at_offset: Vec3,
    pub position_lerp: f32,
    pub rotation_lerp: f32,
    pub keep_targets_visible_radius: Option<f32>,
}

pub enum SceneCameraTargetRef {
    Anchor(SceneAnchorId),
    TargetTag(String),
    LocalPlayer,
}

pub struct SceneCameraAnimationConfig {
    pub on_enter_seconds: f32,
    pub transition_seconds: f32,
    pub easing: SceneCameraEasing,
    pub animate_projection: bool,
}
```

相机目标标记建议：

```rust
#[derive(Component)]
pub struct SceneCameraTarget {
    pub session_id: SceneSessionId,
    pub tag: String,
    pub priority: i32,
}
```

游戏层只需要给实体挂目标标记，framework 负责在当前 session 中解析目标。

## 7. Manifest 草案

固定视角示例：

```ron
camera: Some((
  id: Some("camera.robot_sync_overview"),
  mode: "fixed3d",
  position: Some((0.0, 110.0, 136.0)),
  rotation: Some((-38.9, 0.0, 0.0)),
  projection: Some((kind: "perspective3d", fov_y: 0.78, near: 0.1, far: 300.0)),
  target: Some("anchor.camera_target"),
))
```

跟随视角示例：

```ron
camera: Some((
  id: Some("camera.robot_sync_follow_local"),
  mode: "follow_target",
  position: Some((0.0, 46.0, 58.0)),
  rotation: Some((-42.0, 0.0, 0.0)),
  projection: Some((kind: "perspective3d", fov_y: 0.82, near: 0.1, far: 240.0)),
  follow: Some((
    target: "local_player",
    offset: (0.0, 42.0, 52.0),
    look_at_offset: (0.0, 2.0, 0.0),
    position_lerp: 10.0,
    rotation_lerp: 12.0,
    keep_targets_visible_radius: Some(18.0),
  )),
  animation: Some((
    on_enter_seconds: 0.0,
    transition_seconds: 0.35,
    easing: "smooth_step",
    animate_projection: true,
  )),
))
```

注意：第一阶段如果 manifest 仍只允许单个 `entry.camera`，可以先把固定总览作为默认配置，再通过游戏层命令切换到跟随配置。后续再扩展为 `cameras: []` 多 profile 列表。

## 8. 命令和运行时草案

建议新增轻量命令：

```rust
pub enum SceneCameraCommand {
    SetMode {
        session_id: SceneSessionId,
        mode: SceneCameraMode,
    },
    ApplyConfig {
        session_id: SceneSessionId,
        config: SceneCameraConfig,
        transition_seconds: Option<f32>,
    },
    FollowTarget {
        session_id: SceneSessionId,
        target: SceneCameraTargetRef,
        transition_seconds: Option<f32>,
    },
    ReturnToDefault {
        session_id: SceneSessionId,
        transition_seconds: Option<f32>,
    },
}
```

第一阶段可以先做内部系统和组件，不一定立刻暴露完整命令集。但如果要在玩法中验证固定、跟随和动画切换，至少需要一种 game layer 可调用的切换入口。

## 9. Robot Sync 接入方案

Robot Sync 是第一阶段最合适的验收玩法，因为它已经有固定 3D 相机、动态本地玩家和远端玩家。

建议接入方式：

- `scene.ron` 默认继续使用 `fixed3d` 总览相机。
- 本地玩家 GLB visual 生成后挂 `SceneCameraTarget { tag: "local_player" }`。
- 远端玩家可挂 `SceneCameraTarget { tag: "remote_player" }` 或不挂，取决于第一阶段是否要做保底构图。
- 增加调试配置或按键切换：
  - Overview：固定总览。
  - FollowLocal：跟随本地玩家。
  - AnimatedFollowLocal：从总览平滑过渡到跟随。
- 跟随配置保持较高俯视，避免双客户端联调时完全看不到远端。
- 不把相机位置、FOV 或动画状态写入 replay state。
- 不把相机更新纳入 checksum。

验收时至少覆盖：

- 单客户端进入 Robot Sync 后默认总览可见。
- 本地玩家出现后可切到跟随视角。
- 固定到跟随有平滑过渡。
- 返回大厅后相机、目标标记、Robot Sync visual 都不残留。
- HUD 仍由全局 UI 相机显示。

## 10. 实现顺序

推荐按以下顺序落地：

1. 更新文档，明确职责、配置格式和验收标准。
2. 扩展 `SceneCameraMode`，增加 `FollowTarget`。
3. 扩展 `SceneCameraConfig`、manifest 解析和测试。
4. 增加 `SceneCameraTarget` 组件和目标解析逻辑。
5. 增加 scene camera update system，处理固定、跟随和动画插值。
6. 将 system 注册到 `ScenePlugin`。
7. 在 Robot Sync 本地玩家 visual 上挂相机目标标记。
8. 增加调试切换入口或配置项，确保固定、跟随、动画都可验收。
9. 更新 `docs/scene/场景框架层功能说明.md` 和 `docs/scene/游戏层场景使用说明.md` 中的当前能力描述。
10. 运行 `cargo fmt`、`cargo check` 和相关测试。

## 11. 后续开发任务

以下任务不进入第一阶段完成定义，用于记录摄像机系统后续路线，避免后续设计遗漏。P1-P5 是建议优先级，不等于已经排期。

### P1：多相机 Profile 和相机栈

目标：让一个场景可以声明多个 camera profile，并支持运行时切换。

- 支持 manifest 中配置 `cameras: []` 或等价多 profile 数据结构。
- 支持按 ID 切换总览、跟随、观战、调试和剧情镜头。
- 支持相机 profile fallback，目标缺失或配置不可用时回到默认总览。
- 支持 debug free camera 与 gameplay camera 的明确切换边界。
- 保持 UI 相机独立，不把 HUD 纳入场景相机栈。

### P2：镜头特效系统

目标：把摄像机 transform/projection 类特效标准化，并与渲染/VFX 系统建立事件边界。

- 支持 camera shake、FOV kick、camera roll、zoom pulse。
- 支持 hit stop 期间的镜头冻结或缓动。
- 支持 `CameraImpact`、`CameraFocusChanged`、`CameraEffectIntent` 等意图事件。
- 支持按强度、持续时间、优先级叠加或覆盖镜头效果。
- 不在摄像机系统中直接实现 bloom、vignette、motion blur、color grading 等后处理。

### P3：战斗和剧情镜头

目标：支持战斗、技能、Boss、剧情和交互场景中的镜头调度。

- 支持目标锁定和多目标构图。
- 支持技能释放、命中、Boss 出场、剧情交互等镜头请求。
- 支持轻量 camera path 或 cutscene segment。
- 支持剧情镜头期间的输入阻断、HUD 显隐和返回 gameplay camera。
- 保持剧情镜头不直接耦合具体战斗结算和 authority replay。

### P4：多人、观战和回放镜头

目标：服务联机、观战、录像回放和调试复盘。

- 支持观战目标切换。
- 支持双人或多人可见性约束，避免跟随单个目标时丢失关键远端玩家。
- 支持 replay scrub 时根据历史状态驱动摄像机，不修改 replay 数据。
- 支持 MyServer 或 LAN 联调时快速切回总览。
- 支持摄像机状态诊断输出，便于定位目标缺失、stale session 和构图问题。

### P5：移动端体验、性能和平台适配

目标：把摄像机系统打磨到移动端可长期使用。

- 支持 camera bounds、死区、边缘缓冲和遮挡处理。
- 支持不同 window profile、device scale、横竖屏下的构图参数。
- 支持低端设备降低相机更新复杂度和特效强度。
- 避免高频创建销毁相机实体，优先复用 rig 和配置。
- 增加 Android 真机触控、渲染、HUD 层级和性能验收。

## 12. 测试建议

框架层测试：

- manifest 能解析 `follow_target`、`follow` 和 `animation`。
- `fixed3d` 仍保持原有 transform/projection。
- 跟随目标不存在时保持默认相机或回退 anchor，不 panic。
- 跟随目标移动后，相机 Transform 按 offset 更新。
- 动画 duration 为 0 时立即完成。
- 动画 duration 大于 0 时按 easing 插值。
- 退出场景后 `SceneOwned(session_id)` 相机实体被清理。

游戏层测试：

- Robot Sync 本地玩家 visual 挂相机目标标记。
- stale session 的目标不会被当前相机跟随。
- 切场景或退出后相机目标标记不残留。
- 跟随逻辑不修改 replay state、authority payload 或 checksum。

手动验收：

```powershell
Set-Location project
cargo run -- --window-profile phone-portrait
cargo run -- --window-profile tablet-landscape
```

Robot Sync 手动验收：

```powershell
Set-Location project
$env:MYBEVY_START_SCENE="arena.robot_sync"
cargo run
```

确认：

- 默认总览能看到 arena、边界、出生点和机器人。
- 切换跟随后，本地机器人保持在合理视野范围内。
- 远端机器人在常见距离内仍可被观察，或能快速切回总览。
- HUD 不被场景相机遮挡。
- 返回大厅后无残留相机或目标实体。
