# 游戏层场景使用说明

本文说明 game layer 如何使用 `framework::scene` 接入具体游戏场景。重点面向当前已落地的基础样板场景：从场景表 CSV 注册场景，在 `game_list` 展示入口，进入场景后生成地形、灯光和玩法 HUD。

本文描述当前实现事实和后续扩展约定。样板场景已经打通基础闭环，但不表示已经实现玩家角色、战斗、任务、剧情、碰撞、导航或复杂交互。

## 1. 目标

游戏层场景接入当前已经打通以下闭环：

```text
场景目录表 CSV
-> GameSceneCatalog
-> SceneRegistry 注册
-> game_list 入口
-> SceneCommand::Switch
-> framework 加载 manifest 资源
-> SceneEvent::Entered
-> game layer 实例化场景内容
-> 切换到场景 HUD
-> 返回大厅并清理
```

当前样板场景使用：

```text
scene_id: sample.dungeon_room
kind: dungeon
ui_mode: sample_scene
```

样板场景只验证基础能力，不处理完整玩法逻辑。当前已覆盖：

- CSV 场景目录表。
- 场景注册。
- 大厅入口。
- 场景资源 manifest。
- layout 数据。
- 地形、墙体、装饰物和灯光实例化。
- 场景 HUD 打开和关闭。
- 返回大厅和实体清理。

## 2. 当前目录

当前使用这些路径：

```text
project/assets/game/scenes.csv
project/assets/scenes/sample_dungeon_room/scene.ron
project/assets/scenes/sample_dungeon_room/layout.ron

project/src/game/scenes/
|-- mod.rs
|-- catalog.rs
`-- sample_dungeon_room.rs

project/src/game/screens/gameplay/
|-- mod.rs
|-- sample_scene.rs
`-- touch_ripple.rs
```

职责划分：

- `assets/game/scenes.csv`：游戏层场景目录表。
- `assets/scenes/.../scene.ron`：framework manifest。
- `assets/scenes/.../layout.ron`：game layer 场景内容摆放数据。
- `game/scenes/catalog.rs`：读取和查询 CSV。
- `game/scenes/sample_dungeon_room.rs`：监听样板场景事件、读取 layout、实例化世界内容。
- `game/screens/gameplay/sample_scene.rs`：场景内 HUD。

## 3. 数据分层

### 3.1 CSV 做场景目录

CSV 只表达“有哪些场景可以被游戏层发现、展示和注册”，不表达复杂地图摆放。

当前路径：

```text
project/assets/game/scenes.csv
```

当前字段：

```csv
scene_id,enabled,sort_order,title_key,title_fallback,description_key,description_fallback,kind,manifest_path,layout_path,default_spawn,ui_mode
sample.dungeon_room,true,100,scene.sample_dungeon_room.title,Sample Dungeon Room,scene.sample_dungeon_room.description,Scene directory entry for showcase registration and routing only; prefab transforms live in scene data,dungeon,scenes/sample_dungeon_room/scene.ron,scenes/sample_dungeon_room/layout.ron,spawn.default,sample_scene
```

字段说明：

- `scene_id`：稳定场景 ID。
- `enabled`：是否展示和注册。
- `sort_order`：大厅展示排序。
- `title_key`：i18n 标题 key。
- `title_fallback`：标题 fallback。
- `description_key`：i18n 描述 key。
- `description_fallback`：描述 fallback。
- `kind`：映射到 framework `SceneKind`。
- `manifest_path`：相对 `project/assets/` 的 manifest 路径。
- `layout_path`：相对 `project/assets/` 的 game layer layout 路径。
- `default_spawn`：默认 spawn point。
- `ui_mode`：进入成功后切换到哪个 UI 模式。

当前大厅样板入口仍是固定条目，按钮显示文案来自 `lobby.sample_scene.title`、`lobby.sample_scene.description` 和 `lobby.enter`。CSV 中的 `title_key` / `description_key` 已被读取到 catalog，但当前不驱动大厅 UI，主要为后续动态渲染入口保留。

推荐 Rust 结构：

```rust
#[derive(Resource, Default)]
pub struct GameSceneCatalog {
    entries: Vec<GameSceneEntry>,
}

pub struct GameSceneEntry {
    pub scene_id: SceneId,
    pub enabled: bool,
    pub sort_order: i32,
    pub title_key: String,
    pub title_fallback: String,
    pub description_key: String,
    pub description_fallback: String,
    pub kind: SceneKind,
    pub manifest_path: String,
    pub layout_path: Option<String>,
    pub default_spawn: Option<SceneSpawnPointId>,
    pub ui_mode: GameSceneUiMode,
}
```

当前实现和约定：

- 使用 `csv = "1"` 和 `serde::Deserialize` 解析 CSV。
- 加载失败时记录 warning 并保留空 catalog，避免启动 panic。
- `enabled = false` 的记录可以保留在资源中，但不参与大厅展示和注册。
- 对 `scene_id`、`manifest_path`、`layout_path` 做基础校验。

### 3.2 Manifest 做资源和元数据清单

manifest 交给 framework 读取，负责声明：

- 场景 ID。
- 场景类型。
- 默认 spawn。
- 相机配置。
- Loading 策略。
- 资源 layers。
- spawn points。
- anchors。
- triggers。
- chunks。

推荐路径：

```text
project/assets/scenes/sample_dungeon_room/scene.ron
```

### 3.3 Layout 做实际摆放

layout 是 game layer 自己读取的数据，负责描述地板、墙、装饰物和灯光如何实例化。

推荐路径：

```text
project/assets/scenes/sample_dungeon_room/layout.ron
```

推荐结构：

```rust
#[derive(Deserialize)]
pub struct SampleSceneLayout {
    pub prefabs: Vec<SampleScenePrefab>,
    pub lights: Vec<SampleSceneLight>,
}

#[derive(Deserialize)]
pub struct SampleScenePrefab {
    pub id: String,
    pub asset_path: String,
    pub layer: String,
    pub translation: [f32; 3],
    pub rotation: [f32; 3],
    pub scale: [f32; 3],
}

#[derive(Deserialize)]
pub struct SampleSceneLight {
    pub id: String,
    pub kind: SampleSceneLightKind,
    pub translation: [f32; 3],
    pub rotation: [f32; 3],
    pub color: [f32; 3],
    pub intensity: f32,
    pub range: Option<f32>,
}
```

示例：

```ron
(
  prefabs: [
    (
      id: "floor.center",
      asset_path: "models/scenes/kaykit_dungeon_remastered/floor_tile_large.gltf",

      rotation: [0.0, 0.0, 0.0],
      scale: [1.0, 1.0, 1.0],

    (

      asset_path: "models/props/kaykit_dungeon_remastered/chest.gltf",
      layer: "props",
      translation: [2.0, 0.0, 1.5],
      rotation: [0.0, 35.0, 0.0],
      scale: [1.0, 1.0, 1.0],
    ),
  ],
  lights: [
    (
      id: "sun",
      kind: "directional",
      translation: [0.0, 6.0, 0.0],
      rotation: [-45.0, -25.0, 0.0],
      color: [1.0, 0.94, 0.82],
      intensity: 2500.0,
      range: None,
    ),
    (
      id: "torch.light.01",
      kind: "point",
      translation: [2.5, 1.8, 2.5],
      rotation: [0.0, 0.0, 0.0],
      color: [1.0, 0.58, 0.28],
      intensity: 350.0,
      range: Some(6.0),
    ),
  ],

```

layout 放在 game layer 的原因：

- prefab 摆放会跟具体项目资源强绑定。
- 灯光、装饰物、怪物、交互点都属于具体游戏内容。
- 后续可能替换为编辑器导出格式，不应污染 framework。
- framework manifest 应保持稳定、轻量和通用。

## 5. 注册流程

启动时：

1. 读取 `project/assets/game/scenes.csv`。
2. 转换为 `GameSceneCatalog`。
3. 按 `enabled` 过滤。
4. 把 `kind` 转换为 `SceneKind`。
5. 调用 `SceneRegistry::register_manifest_scene`。
6. 保存 catalog 资源；当前大厅样板入口仍是固定条目，后续可改为按 catalog 动态渲染。

伪代码：

```rust
fn register_game_scenes(
    catalog: Res<GameSceneCatalog>,
    mut registry: ResMut<SceneRegistry>,
) {
    for entry in catalog.enabled_entries() {
        if let Err(error) = registry.register_manifest_scene(
            entry.scene_id.clone(),
            entry.kind,
            entry.manifest_path.clone(),
        ) {
            warn!("failed to register game scene {}: {error}", entry.scene_id);
        }
    }
}
```

建议策略：

- 单个场景注册失败不应让整个游戏崩溃。
- 启动日志中应明确失败场景 ID 和 manifest path。
- 样板场景可额外定义常量，减少字符串散落：

```rust
pub const SAMPLE_DUNGEON_ROOM_SCENE_ID: &str = "sample.dungeon_room";
```

## 6. 大厅入口

当前 `game_list` 里已有 Touch Ripple 的入口和样板场景固定入口：

```text
Sample Scene
Dungeon room scene prototype
[Enter]
```

当前固定入口的显示文案来自 `lobby.sample_scene.*`，不是 CSV 行里的 `scene.sample_dungeon_room.*`。如果进入失败，大厅会清理 pending 状态并发送 `lobby.sample_scene.toast.failed` toast；该 toast 当前依赖 key fallback，后续可以补进首包 i18n 表。

点击按钮后当前发送：

```rust
scene_commands.write(SceneCommand::Switch(SceneSwitchRequest::new(
    SAMPLE_DUNGEON_ROOM_SCENE_ID,
)));
```

当前实现不会在点击按钮后立刻切 HUD，而是按下面时序：

```text
点击大厅按钮
-> 发送 SceneCommand::Switch
-> framework 加载 manifest 和 required asset
-> 收到 SceneEvent::Entered
-> game layer 实例化 layout 内容
-> 发送 GameRouteCommand::ChangeMode(AppUiMode::SampleScene)
```

这样如果 manifest 或资源加载失败，玩家仍停留在大厅，UI 不会进入一个没有场景内容的 HUD。

后续可把 `game_list` 从固定条目改为读取 `GameSceneCatalog` 动态渲染：

- 按 `sort_order` 排序。
- 只显示 `enabled = true`。
- 使用 `title_key` 和 `description_key` 做 i18n。
- 按 `ui_mode` 决定进入成功后的 HUD。

```rust
    Lobby,
    WanfaTouchRipple,

    SampleScene,
}
```

`ui_ids.rs` 中已经包含：

- `OWNER_SAMPLE_SCENE`
- `PANEL_SAMPLE_SCENE_HUD`

HUD 放在：

```text
project/src/game/screens/gameplay/sample_scene.rs
```

HUD 行为：

- `OnEnter(AppUiMode::SampleScene)` spawn。
- `OnExit(AppUiMode::SampleScene)` 自动 despawn。
- 显示场景标题、状态文本、返回大厅按钮。
- 返回按钮发送 `SceneCommand::Exit` 和 `GameRouteCommand::ChangeMode(AppUiMode::Lobby)`。

HUD 不要挂到 `SceneRoot` 下。HUD 属于 UI 框架，生命周期由 `AppUiMode` 管理。

## 8. 进入场景后的内容实例化

样板场景内容由 `game/scenes/sample_dungeon_room.rs` 监听 `SceneEvent::Entered` 后生成。

伪代码：

```rust
fn handle_sample_scene_entered(
    mut commands: Commands,
    asset_server: Res<AssetServer>,
    mut scene_events: MessageReader<SceneEvent>,
    layer_roots: Query<(Entity, &SceneLayerRoot)>,
    runtime_roots: Query<(Entity, &SceneRuntimeRoot)>,
    mut route_commands: MessageWriter<GameRouteCommand>,
) {
    for event in scene_events.read() {
        let SceneEvent::Entered(entered) = event else {
            continue;
        };

        if entered.scene_id.as_str() != SAMPLE_DUNGEON_ROOM_SCENE_ID {
            continue;
        }

        // 1. 读取 layout.ron
        // 2. 找到当前 session 的 layer root 或 runtime root
        // 3. spawn glTF prefab
        // 4. spawn lights
        // 5. 所有实体挂 SceneOwned(entered.session_id.clone())
        // 6. 切换到 SampleScene HUD
        route_commands.write(GameRouteCommand::ChangeMode(AppUiMode::SampleScene));
    }
}
```

实例化 glTF 时注意：

- Bevy 原生 glTF scene 通常通过 `SceneRoot(handle)` 或相关 bundle 实例化。
- MyBevy 的 `framework::scene::SceneRoot` 与 Bevy 的 `SceneRoot` 名称可能冲突，导入时建议使用别名。
- 实例化实体应挂 `SceneOwned(session_id)`。
- 实体应挂到 framework 的 `SceneLayerRoot` 或 `SceneRuntimeRoot` 下。
- 每个实体设置可读 `Name`，方便调试。

命名示例：

```text
SampleDungeonPrefab(floor.center)
SampleDungeonPrefab(chest.01)
SampleDungeonLight(sun)
SampleDungeonLight(torch.light.01)
```

## 9. 地形和摆放建议

当前样板场景是一个小型 dungeon room：

- 地板：`floor_tile_large.gltf` 拼 5x5。
- 墙体：`wall.gltf` 包围四周。
- 墙角：`wall_corner.gltf` 放四角。
- 入口：`wall_doorway.gltf` 放南侧。
- 装饰：`chest.gltf`、`barrel_large.gltf`、`box_large.gltf`。
- 光源装饰：`torch_lit.gltf`。
- 灯光：一个 `DirectionalLight` 加两个 `PointLight`。

移动端注意：

- 第一版不要放太多实时点光。
- 不要开启大量阴影。
- 不要一次 spawn 太多独立 glTF 实例。
- 当前优先验证结构和流程，不追求美术完整度。

## 10. 灯光和环境

样板场景 layout 当前包含 light 定义。进入场景后按 light kind 生成：

- `DirectionalLight`
- `PointLight`

建议：

- 主方向光负责整体可见性。
- 点光只用于火把或重点装饰。
- 所有灯光都挂 `SceneOwned(session_id)`。
- 点光数量保持很少。
- 亮度先以桌面可见为准，再做移动端调优。

如果修改 `ClearColor`，需要考虑返回大厅后是否恢复。更稳的方式是让 HUD 页面或场景系统明确设置当前模式需要的背景色。

## 11. 退出和清理

返回大厅时当前顺序：

```text
HUD 返回按钮点击
-> SceneCommand::Exit
-> GameRouteCommand::ChangeMode(AppUiMode::Lobby)
-> framework 清理 SceneOwned 实体
-> UI 框架关闭 SampleScene HUD
```

必须验证：

- 地形实体被清理。
- 灯光实体被清理。
- 场景相机被清理。
- trigger 被清理。
- HUD 被清理。
- 再次进入不会出现重复实体。

如果有全局资源不应该随场景退出清理，不要挂 `SceneOwned`。

## 12. 错误处理

游戏层应监听 `SceneEvent::Failed`：

```rust
fn handle_scene_failed(
    mut events: MessageReader<SceneEvent>,
    mut overlay_commands: MessageWriter<UiOverlayCommand>,
) {
    for event in events.read() {
        let SceneEvent::Failed(failure) = event else {
            continue;
        };

        // 显示面向玩家的 toast 或 modal
        // 日志记录 failure.log_description()
    }
}
```

建议行为：

- 如果从大厅进入样板场景失败，停留在大厅。
- 显示简短错误提示，例如“场景加载失败”。
- 日志记录 scene_id、manifest path、asset path 和错误分类。
- 不直接把内部路径显示给玩家。

## 13. 和现有 Touch Ripple 的关系

Touch Ripple 当前是单界面玩法，已有自己的 HUD 和 authority 输入回放。

样板场景接入没有把 Touch Ripple 改造成场景系统的一部分。当前样板场景是独立入口，用来验证 scene framework 的世界内容流程。

需要保持：

- 登录流程不变。
- 大厅流程不变。
- Touch Ripple 入口不变。
- UI Gallery 入口不变。
- Touch Ripple 的 `AppUiMode::WanfaTouchRipple` 不被样板场景影响。

## 16. 常见问题

### 16.1 为什么 CSV 不直接写 prefab 坐标？

CSV 适合扁平目录，不适合复杂嵌套数据。prefab、灯光、触发器、装饰物和后续怪物刷点都需要结构化字段，用 RON 更清晰，也和项目现有依赖匹配。

### 16.2 为什么 manifest 声明了 glTF 还要 layout？

manifest 的职责是让 framework 知道哪些资源需要加载、哪些 spawn/anchor/trigger 存在。layout 的职责是告诉 game layer 如何把具体 prefab 放到世界里。这两个职责分开后，framework 能保持通用。

### 16.3 为什么 HUD 不挂到 SceneRoot？

HUD 是 UI 层内容，应由 UI 框架管理。`SceneRoot` 是世界内容根，用于地形、相机、灯光、trigger 和动态场景实体。把 HUD 挂到世界 root 会混淆清理边界和输入路由。

### 16.4 进入失败时应该切换 UI 吗？

不建议。进入命令发出后应等待 `SceneEvent::Entered`，成功后再切 HUD。失败时保持大厅界面，并显示 toast 或 modal。

### 16.5 场景内玩家角色什么时候加？

第一版样板场景不需要玩家角色。后续可以在 `SceneEvent::Ready` 或 `Entered` 后，根据 spawn point 生成角色实体，并接入输入、动画和 authority。
