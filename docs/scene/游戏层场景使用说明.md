# 游戏层场景使用说明

本文说明 game layer 如何使用 `framework::scene` 接入具体游戏场景。当前覆盖两个已落地场景：`sample.dungeon_room` 基础样板场景，以及 `arena.robot_sync` 机器人帧同步正式小场景。

本文描述当前实现事实和后续扩展约定。样板场景已经打通基础闭环，但不表示已经实现玩家角色、战斗、任务、剧情、碰撞、导航或复杂交互。

## 1. 目标

游戏层场景接入当前已经打通以下闭环：

```text
场景目录表 CSV
-> GameSceneCatalog
-> SceneRegistry 注册
-> game_list 入口
-> SceneCommand::Switch
-> framework 加载 manifest 资源
-> SceneEvent::Entered
-> game layer 实例化场景内容
-> 切换到场景 HUD
-> 返回大厅并清理
```

当前样板场景使用：

```text
scene_id: sample.dungeon_room
kind: dungeon
ui_mode: sample_scene
```

当前机器人帧同步场景使用：

```text
scene_id: arena.robot_sync
kind: arena
ui_mode: robot_sync_scene
```

样板场景只验证基础能力，不处理完整玩法逻辑。当前已覆盖：

- CSV 场景目录表。
- 场景注册。
- 大厅入口。
- 场景资源 manifest。
- layout 数据。
- 地形、墙体、装饰物和灯光实例化。
- 场景 HUD 打开和关闭。
- 返回大厅和实体清理。

`arena.robot_sync` 在同一套场景目录和生命周期上额外验证：

- 500x500 轻量 arena 场地、边界、网格和出生点。
- local、LAN 和 MyServer authority 启动。
- MyServer `robot_sync_room` policy 加房、ready、start。
- 本地 bot 发送 `robot_move`。
- 消费 `AuthorityEvent::FrameApplied` 做确定性 replay。
- 机器人视觉实体和 Robot Sync HUD。
- 双客户端同帧 checksum / fixed 坐标一致性验收。

## 3. 数据分层

### 3.1 CSV 做场景目录

CSV 只表达“有哪些场景可以被游戏层发现、展示和注册”，不表达复杂地图摆放。

当前路径：

```text
project/assets/game/scenes.csv
```

当前字段：

```csv
scene_id,enabled,sort_order,title_key,title_fallback,description_key,description_fallback,kind,manifest_path,layout_path,default_spawn,ui_mode,music_clip_id,music_path,music_volume,music_fade_in_seconds,music_exit_fade_out_seconds
sample.dungeon_room,true,100,scene.sample_dungeon_room.title,Sample Dungeon Room,scene.sample_dungeon_room.description,Scene directory entry for showcase registration and routing only; prefab transforms live in scene data,dungeon,scenes/sample_dungeon_room/scene.ron,scenes/sample_dungeon_room/layout.ron,spawn.default,sample_scene,music.sample_dungeon_room,audio/music/stealth_bass_loop.wav,0.35,0.5,0.4
arena.robot_sync,true,200,scene.robot_sync_arena.title,Robot Sync Arena,scene.robot_sync_arena.description,500x500 robot frame sync arena,arena,scenes/robot_sync_arena/scene.ron,scenes/robot_sync_arena/layout.ron,spawn.robot_a,robot_sync_scene,,,,,
```

字段说明：

- `scene_id`：稳定场景 ID。
- `enabled`：是否展示和注册。
- `sort_order`：大厅展示排序。
- `title_key`：i18n 标题 key。
- `title_fallback`：标题 fallback。
- `description_key`：i18n 描述 key。
- `description_fallback`：描述 fallback。
- `kind`：映射到 framework `SceneKind`。
- `manifest_path`：相对 `project/assets/` 的 manifest 路径。
- `layout_path`：相对 `project/assets/` 的 game layer layout 路径。
- `default_spawn`：默认 spawn point。
- `ui_mode`：进入成功后切换到哪个 UI 模式。
- `music_clip_id`：可选背景音乐 clip ID；填写音乐任一字段时必须同时填写 `music_clip_id` 和 `music_path`。
- `music_path`：可选背景音乐资源路径，写相对 `project/assets/` 的路径，例如 `audio/music/stealth_bass_loop.wav`。
- `music_volume`：可选背景音乐音量，有限非负数；为空时使用音频框架默认音量。
- `music_fade_in_seconds`：可选进入场景音乐淡入秒数，有限非负数。
- `music_exit_fade_out_seconds`：可选退出场景按 scene session scope 停止时的淡出秒数，有限非负数。

当前大厅的样板场景和 Robot Sync 入口仍是固定条目，按钮显示文案来自 `lobby.sample_scene.*`、`lobby.robot_sync_scene.*` 和 `lobby.enter`。CSV 中的 `title_key` / `description_key` 已被读取到 catalog，但当前不驱动大厅 UI，主要为后续动态渲染入口保留。

推荐 Rust 结构：

```rust
#[derive(Resource, Default)]
pub struct GameSceneCatalog {
    entries: Vec<GameSceneEntry>,
}

pub struct GameSceneEntry {
    pub scene_id: SceneId,
    pub enabled: bool,
    pub sort_order: i32,
    pub title_key: String,
    pub title_fallback: String,
    pub description_key: String,
    pub description_fallback: String,
    pub kind: SceneKind,
    pub manifest_path: String,
    pub layout_path: Option<String>,
    pub default_spawn: Option<SceneSpawnPointId>,
    pub ui_mode: GameSceneUiMode,
    pub music: Option<GameSceneMusicConfig>,
}

pub struct GameSceneMusicConfig {
    pub clip_id: AudioClipId,
    pub path: String,
    pub volume: Option<f32>,
    pub fade_in_seconds: Option<f32>,
    pub exit_fade_out_seconds: Option<f32>,
}
```

当前实现和约定：

- 使用 `csv = "1"` 和 `serde::Deserialize` 解析 CSV。
- 加载失败时记录 warning 并保留空 catalog，避免启动 panic。
- `enabled = false` 的记录可以保留在资源中，但不参与大厅展示和注册。
- 对 `scene_id`、`manifest_path`、`layout_path` 做基础校验。
- 音乐字段为空时不注册 BGM；填写音乐任一字段时校验 clip/path 组合完整，数值字段必须为有限非负数。

### 3.2 Manifest 做资源和元数据清单

- 场景 ID。
- 场景类型。
- 默认 spawn。
- 相机配置。
- Loading 策略。
- 资源 layers。
- spawn points。
- anchors。
- triggers。
- chunks。

推荐路径：


### 3.3 Layout 做实际摆放

layout 是 game layer 自己读取的数据，负责描述地板、墙、装饰物和灯光如何实例化。

推荐路径：

```text
project/assets/scenes/sample_dungeon_room/layout.ron
```

推荐结构：

```rust
#[derive(Deserialize)]
pub struct SampleSceneLayout {
    pub prefabs: Vec<SampleScenePrefab>,
    pub lights: Vec<SampleSceneLight>,
}

#[derive(Deserialize)]

    pub id: String,
    pub asset_path: String,
    pub layer: String,
```

layout 放在 game layer 的原因：

- prefab 摆放会跟具体项目资源强绑定。

- 后续可能替换为编辑器导出格式，不应污染 framework。
- framework manifest 应保持稳定、轻量和通用。

## 5. 注册流程

启动时：

1. 读取 `project/assets/game/scenes.csv`。
2. 转换为 `GameSceneCatalog`。
3. 按 `enabled` 过滤。
4. 把 `kind` 转换为 `SceneKind`。
5. 调用 `SceneRegistry::register_manifest_scene`。
6. 保存 catalog 资源；当前大厅样板入口仍是固定条目，后续可改为按 catalog 动态渲染。

伪代码：

```rust
fn register_game_scenes(
    catalog: Res<GameSceneCatalog>,
    mut registry: ResMut<SceneRegistry>,
) {
    for entry in catalog.enabled_entries() {
        if let Err(error) = registry.register_manifest_scene(
            entry.scene_id.clone(),
            entry.kind,
            entry.manifest_path.clone(),
        ) {
            warn!("failed to register game scene {}: {error}", entry.scene_id);
        }
    }
}
```

建议策略：

- 单个场景注册失败不应让整个游戏崩溃。
- 启动日志中应明确失败场景 ID 和 manifest path。
- 具体场景可额外定义常量，减少字符串散落：

```rust
pub const SAMPLE_DUNGEON_ROOM_SCENE_ID: &str = "sample.dungeon_room";
pub const ROBOT_SYNC_ARENA_SCENE_ID: &str = "arena.robot_sync";
```

## 6. 大厅入口

当前 `game_list` 里已有 Touch Ripple、样板场景和 Robot Sync 的固定入口：

```text
Sample Scene
Dungeon room scene prototype
[Enter]

Robot Sync
500x500 robot frame sync arena
[Enter]
```

当前固定入口的显示文案来自 `lobby.sample_scene.*` 和 `lobby.robot_sync_scene.*`，不是 CSV 行里的 `scene.*` 字段。如果进入失败，大厅会清理对应 pending 状态并发送 toast；toast 当前依赖 key fallback，后续可以补进首包 i18n 表。

点击按钮后当前发送：

```rust
scene_commands.write(SceneCommand::Switch(SceneSwitchRequest::new(
    SAMPLE_DUNGEON_ROOM_SCENE_ID,
)));

scene_commands.write(SceneCommand::Switch(SceneSwitchRequest::new(
    ROBOT_SYNC_ARENA_SCENE_ID,
)));
```

当前实现不会在点击按钮后立刻切 HUD，而是按下面时序：

```text
点击大厅按钮
-> 发送 SceneCommand::Switch
-> framework 加载 manifest 和 required asset
-> 收到 SceneEvent::Entered
-> game layer 实例化 layout 内容
-> 发送 GameRouteCommand::ChangeMode(AppUiMode::SampleScene 或 AppUiMode::RobotSyncScene)
```

这样如果 manifest 或资源加载失败，玩家仍停留在大厅，UI 不会进入一个没有场景内容的 HUD。

后续可把 `game_list` 从固定条目改为读取 `GameSceneCatalog` 动态渲染：

- 按 `sort_order` 排序。
- 只显示 `enabled = true`。
- 使用 `title_key` 和 `description_key` 做 i18n。
- 按 `ui_mode` 决定进入成功后的 HUD。

```text

```

HUD 行为：

- `OnEnter(AppUiMode::SampleScene)` spawn。
- `OnExit(AppUiMode::SampleScene)` 自动 despawn。
- `OnEnter(AppUiMode::RobotSyncScene)` spawn Robot Sync HUD，`OnExit(AppUiMode::RobotSyncScene)` 自动 despawn。

- Robot Sync HUD 显示 room、player、authority 状态、frame、机器人数量、本地 fixed/world 坐标和返回大厅按钮。
- 返回按钮发送 `SceneCommand::Exit` 和 `GameRouteCommand::ChangeMode(AppUiMode::Lobby)`。

HUD 不要挂到 `SceneRoot` 下。HUD 属于 UI 框架，生命周期由 `AppUiMode` 管理。

## 8. 进入场景后的内容实例化

样板场景内容由 `game/scenes/sample_dungeon_room.rs` 监听 `SceneEvent::Entered` 后生成。

伪代码：

```rust
fn handle_sample_scene_entered(
    mut commands: Commands,
    asset_server: Res<AssetServer>,
    mut scene_events: MessageReader<SceneEvent>,
    layer_roots: Query<(Entity, &SceneLayerRoot)>,
    runtime_roots: Query<(Entity, &SceneRuntimeRoot)>,
    mut route_commands: MessageWriter<GameRouteCommand>,
) {
    for event in scene_events.read() {
        let SceneEvent::Entered(entered) = event else {
            continue;
        };

        if entered.scene_id.as_str() != SAMPLE_DUNGEON_ROOM_SCENE_ID {
            continue;
        }

        // 1. 读取 layout.ron
        // 2. 找到当前 session 的 layer root 或 runtime root
        // 3. spawn glTF prefab
        // 4. spawn lights
        // 5. 所有实体挂 SceneOwned(entered.session_id.clone())
        // 6. 切换到 SampleScene HUD
        route_commands.write(GameRouteCommand::ChangeMode(AppUiMode::SampleScene));
    }
}
```

实例化 glTF 时注意：

- Bevy 原生 glTF scene 通常通过 `SceneRoot(handle)` 或相关 bundle 实例化。
- MyBevy 的 `framework::scene::SceneRoot` 与 Bevy 的 `SceneRoot` 名称可能冲突，导入时建议使用别名。
- 实例化实体应挂 `SceneOwned(session_id)`。
- 实体应挂到 framework 的 `SceneLayerRoot` 或 `SceneRuntimeRoot` 下。
- 每个实体设置可读 `Name`，方便调试。

命名示例：

```text
SampleDungeonPrefab(floor.center)
SampleDungeonPrefab(chest.01)
SampleDungeonLight(sun)
SampleDungeonLight(torch.light.01)
```

## 9. 地形和摆放建议

当前样板场景是一个小型 dungeon room：

- 地板：`floor_tile_large.gltf` 拼 5x5。
- 墙体：`wall.gltf` 包围四周。
- 墙角：`wall_corner.gltf` 放四角。
- 入口：`wall_doorway.gltf` 放南侧。
- 装饰：`chest.gltf`、`barrel_large.gltf`、`box_large.gltf`。
- 光源装饰：`torch_lit.gltf`。
- 灯光：一个 `DirectionalLight` 加两个 `PointLight`。

移动端注意：

- 第一版不要放太多实时点光。
- 不要开启大量阴影。
- 不要一次 spawn 太多独立 glTF 实例。
- 当前优先验证结构和流程，不追求美术完整度。

## 10. 灯光和环境

样板场景 layout 当前包含 light 定义。进入场景后按 light kind 生成：

- `DirectionalLight`
- `PointLight`

建议：

- 主方向光负责整体可见性。
- 点光只用于火把或重点装饰。
- 所有灯光都挂 `SceneOwned(session_id)`。
- 点光数量保持很少。
- 亮度先以桌面可见为准，再做移动端调优。

如果修改 `ClearColor`，需要考虑返回大厅后是否恢复。更稳的方式是让 HUD 页面或场景系统明确设置当前模式需要的背景色。

- 灯光实体被清理。

## 12. `arena.robot_sync` 接入说明

`arena.robot_sync` 是当前第一版机器人帧同步正式场景。它复用 scene framework 的 manifest、root、相机和清理能力，但同步规则全部放在 game layer 和 MyServer 对应 room policy 中。

数据文件：

```text
project/assets/game/scenes.csv
project/assets/scenes/robot_sync_arena/scene.ron
project/assets/scenes/robot_sync_arena/layout.ron
```

CSV 行：

- `scene_id=arena.robot_sync`
- `enabled=true`
- `kind=arena`
- `manifest_path=scenes/robot_sync_arena/scene.ron`
- `layout_path=scenes/robot_sync_arena/layout.ron`
- `default_spawn=spawn.robot_a`
- `ui_mode=robot_sync_scene`
- 音乐字段当前为空，不注册场景 BGM。

manifest 当前职责：

- 声明 `SceneId("arena.robot_sync")` 和 `kind="arena"`。
- 使用 `fixed3d` 透视相机，目标 anchor 为 `anchor.camera_target`，相机顺序低于全局 UI 摄像机。
- 声明 `arena`、`grid`、`spawns`、`robots` 四个轻量 layer，当前都不加载外部重型资源。
- 声明 `spawn.robot_a`、`spawn.robot_b` 和场地中心、相机目标、四角 bounds anchors；两个 Robot Sync 验收出生点分别位于场地左右两侧。

layout 当前职责：

- 声明 500x500 arena，逻辑范围为 `x/y = -250..250`。
- 声明 50 单位网格、4 单位边界厚度、两个出生点和机器人半径。
- 声明场地、网格、边界和出生点颜色。

场景内容实例化：

- `project/src/game/scenes/robot_sync_arena.rs` 在 `SceneEvent::Entered` 且 scene id 匹配时读取 layout。
- 找到当前 session 的 `SceneRuntimeRoot` 后，在其下生成一个 arena content 根实体。
- 当前 3D 场地会复用 `floor_tile_large.gltf` 生成 9 个放大地板拼片覆盖 `-250..250` 逻辑范围，并生成下沉 primitive 场地底座、4 条边界、22 条网格线、2 个贴地 spawn marker 和 1 个 `DirectionalLight`。
- 所有实体都挂 `SceneOwned(session_id)` 和 robot sync arena 标记，重复收到同一 session 的 Entered 不重复生成。

玩法模块职责：

- `project/src/game/features/robot_sync/config.rs` 读取环境变量，默认 `ROBOT_SYNC_AUTHORITY_MODE=local`，MyServer policy 默认 `robot_sync_room`。
- `plugin.rs` 响应 `arena.robot_sync` 的进入/退出，启动或离开 local、LAN 或 MyServer authority；MyServer 模式按登录、加房、ready、start 推进。
- `bot.rs` / `plugin.rs` 生成本地输入，发送 `robot_move` 到 `session.frame_id + input_delay_frames`；默认 `ROBOT_SYNC_INPUT_MODE=bot`，也支持 `ROBOT_SYNC_INPUT_MODE=manual` 用 `WASD` 或方向键控制本地机器人，松开按键会发送停止输入。默认 `input_delay_frames=2`，当前 bot/manual speed 默认 `10000`，匹配 MyServer 当前校验范围。
- `sync.rs` 消费 `AuthorityEvent::Snapshot` 和 `AuthorityEvent::FrameApplied`，按 fixed units 做 deterministic replay；位置只在权威帧推进，不使用 Bevy `Time`。
- `visual.rs` 按 replay state 为每个 player 生成或移动一个轻量 2D robot entity，本地玩家有尺寸和层级区分。
- `hud.rs` 提供 UI snapshot/format helper，HUD 不直接读取内部 HashMap 字段。

MyServer 模式依赖：

- MyServer 必须显式支持 `robot_sync_room` policy 和 `RobotSyncRoomLogic`。
- 客户端必须使用同一 room id，两个客户端使用不同 `AUTHORITY_PLAYER_ID` / `AUTHORITY_MYSERVER_GUEST_ID`。
- MyServer 只校验并转发 `robot_move`，不计算或广播机器人坐标，也不发送 `MovementSnapshotPush` 覆盖机器人 replay。

关键日志：

- 发送输入：自动 bot 日志为 `sending robot sync bot input`，手动键盘日志为 `sending robot sync manual input`，都包含 `player_id`、`target_frame`、`seq`、`botTick`、`dirX`、`dirY`、`speed`。
- 回放 telemetry：`robot sync frame applied`，包含 `frame_id`、`robot_count`、`checksum`、`robots`。
- MyServer 加房和错误日志应能看到 `room_id`、`policy_id`、`player_id`、`guest_id`、transport、错误码和原因。

手动启动：

```powershell

$env:MYBEVY_START_SCENE="arena.robot_sync"
cargo run -- --window-profile phone-small --window-scale 50%
```

双客户端 MyServer 联调：

```powershell

# C:\project\mybevy
.\scripts\start-robot-sync-two-clients.ps1 -SkipBuild -HostAddress 127.0.0.1 -Port 17002

# 默认 A 端手动控制，B 端静止观察；如需自动 bot，可显式传 -InputModeB bot
```

Robot Sync 双客户端脚本只设置 `ROBOT_SYNC_AUTHORITY_MODE`，并在 client launcher 中清空继承的 `AUTHORITY_DEV_MODE`。不要额外给这组窗口设置 `AUTHORITY_DEV_MODE`，否则全局 dev authority 会和场景内 Robot Sync authority 重复启动，LAN 模式下可能抢占同一端口并导致 B 端长期 `pending`。

## 17. 常见问题

### 17.1 为什么 CSV 不直接写 prefab 坐标？

CSV 适合扁平目录，不适合复杂嵌套数据。prefab、灯光、触发器、装饰物和后续怪物刷点都需要结构化字段，用 RON 更清晰，也和项目现有依赖匹配。

### 17.2 为什么 manifest 声明了 glTF 还要 layout？

manifest 的职责是让 framework 知道哪些资源需要加载、哪些 spawn/anchor/trigger 存在。layout 的职责是告诉 game layer 如何把具体 prefab 放到世界里。这两个职责分开后，framework 能保持通用。

### 17.3 为什么 HUD 不挂到 SceneRoot？

HUD 是 UI 层内容，应由 UI 框架管理。`SceneRoot` 是世界内容根，用于地形、相机、灯光、trigger 和动态场景实体。把 HUD 挂到世界 root 会混淆清理边界和输入路由。

### 17.4 进入失败时应该切换 UI 吗？

不建议。进入命令发出后应等待 `SceneEvent::Entered`，成功后再切 HUD。失败时保持大厅界面，并显示 toast 或 modal。

### 17.5 场景内玩家角色什么时候加？

第一版样板场景不需要玩家角色。后续可以在 `SceneEvent::Ready` 或 `Entered` 后，根据 spawn point 生成角色实体，并接入输入、动画和 authority。

### 17.6 为什么 Robot Sync 不用 `MovementSnapshotPush`？

当前第一版目标是验证输入帧同步和客户端确定性 replay。MyServer `robot_sync_room` 只校验并转发 `robot_move`，客户端只消费 `FrameApplied` 推进机器人位置。服务端权威坐标、回滚预测和 `MovementSnapshotPush` 属于后续独立设计，不应混入当前最小闭环。
