# 场景框架层功能说明

本文说明 `project/src/framework/scene/` 当前提供的框架层 Scene 能力、模块边界、运行流程和接入约定。这里的 Scene 指 MyBevy 自定义的场景运行时，不等同于 Bevy 原生 `Scene` asset，也不等同于 glTF 文件内部的 scene。

本文只描述 framework layer。具体游戏场景 ID、场景表、样板房间、地形摆放、玩法 HUD、角色、战斗、任务和剧情都应放在 game layer。

## 1. 定位

`framework::scene` 的目标是提供一套可复用的场景生命周期和资源边界，让游戏层可以用统一方式进入、退出、切换和查询场景。

框架层负责：

- 管理场景注册表。
- 接收场景进入、退出、切换、重载等命令。
- 维护当前场景运行状态。
- 从首包资源读取场景 manifest。
- 跟踪 manifest 中声明的资源加载状态。
- 根据场景定义创建和清理场景根实体。
- 创建基础场景相机。
- 建立 spawn point 和 anchor 查询索引。
- 创建通用 trigger 实体并派发 trigger 事件。
- 维护 chunk/streaming 元数据状态。
- 通过 UI 框架显示和关闭全局 Loading。
- 提供 authority ready 的框架层边界。
- 提供调试配置和诊断查询入口。

框架层不负责：

- 实例化具体地图 prefab、地形、墙体、装饰物和角色。
- 解释某个 glTF 节点代表什么玩法对象。
- 生成具体碰撞体、导航网格、战斗对象、任务对象或剧情脚本。
- 决定大厅里展示哪些场景入口。
- 打开具体玩法 HUD。
- 处理服务端房间、匹配、角色同步或帧同步业务规则。
- 处理后续下载资源的真实缓存、校验和 CDN 流程。

## 2. 当前模块

当前场景框架位于：

```text
project/src/framework/scene/
|-- mod.rs
|-- authority.rs
|-- camera.rs
|-- command.rs
|-- debug.rs
|-- event.rs
|-- id.rs
|-- lifecycle.rs
|-- loading.rs
|-- manifest.rs
|-- plugin.rs
|-- prelude.rs
|-- registry.rs
|-- root.rs
|-- spawn.rs
|-- streaming.rs
`-- trigger.rs
```

模块职责：

- `plugin.rs`：`ScenePlugin` 入口，注册消息、资源和系统。
- `command.rs`：场景命令类型，例如 `SceneCommand::Enter`、`Exit`、`Switch`。
- `event.rs`：场景事件类型，例如 `Resolving`、`LoadProgress`、`Entered`、`Failed`。
- `lifecycle.rs`：场景运行状态机、进入/退出/切换流程、ready 状态。
- `registry.rs`：场景注册表、场景定义和 `SceneKind`。
- `manifest.rs`：首包 RON manifest 数据结构、读取和校验。
- `loading.rs`：asset 加载跟踪、Loading UI 联动。
- `root.rs`：`SceneRoot`、`SceneLayerRoot`、`SceneRuntimeRoot`、`SceneOwned`。
- `camera.rs`：基础 2D/3D 场景相机配置和生成。
- `spawn.rs`：spawn point、anchor 和查询索引。
- `trigger.rs`：通用触发器 manifest、实体、命令和事件。
- `streaming.rs`：zone、region、chunk 元数据和 streaming 状态。
- `authority.rs`：framework 层 ready 请求和 ready 状态。
- `debug.rs`：环境变量配置、调试快照和诊断信息。
- `prelude.rs`：对外集中导出游戏层常用类型。

## 3. 插件入口

游戏或上层模块需要显式添加 `ScenePlugin`：

```rust
use crate::framework::scene::prelude::ScenePlugin;

app.add_plugins(ScenePlugin);
```

`ScenePlugin` 当前会注册：

- `SceneCommand`
- `SceneEvent`
- `SceneAuthorityReadyRequest`
- `SceneAuthorityReadyStatus`
- `SceneTriggerCommand`
- `SceneTriggerEvent`
- `SceneRuntime`
- `SceneAssetLoadQueue`
- `SceneLoadingUiConfig`
- `SceneLoadingUiState`
- `SceneRegistry`
- `SceneSpawnRegistry`
- `SceneStreamingState`
- `SceneStreamingDriverConfig`
- `SceneDebugConfig`

并在 `Update` 阶段串行执行：

```text
process_scene_lifecycle_commands
poll_scene_asset_loads
process_scene_trigger_commands
detect_scene_triggers
update_scene_streaming_driver
sync_scene_loading_ui
```

这意味着同一帧中如果发送多个进入/退出/切换命令，生命周期系统会按当前实现聚合处理最后一个主生命周期请求。游戏层不要依赖“同一帧连续进入多个场景”的行为。

## 4. 核心概念

### 4.1 SceneId

`SceneId` 是稳定场景 ID。业务、服务端协议、存档和 UI 入口都应引用场景 ID，而不是直接引用资源路径。

推荐命名：

```text
sample.dungeon_room
lobby.main
world.capital
dungeon.forest_001
arena.test_small
```

ID 应保持稳定，不建议携带内容版本号、平台后缀或临时路径。

### 4.2 SceneKind

`SceneKind` 描述场景类型，当前包含：

- `Boot`
- `Ui`
- `Lobby`
- `Gameplay`
- `Dungeon`
- `World`
- `Arena`
- `Dev`

`SceneKind` 主要用于加载策略、默认行为和后续扩展。当前不会自动生成玩法逻辑。

### 4.3 SceneRuntime

`SceneRuntime` 是场景运行状态资源，记录：

- 当前 active session。
- 当前 pending session。
- 当前 ready 信息。
- 当前生命周期状态。
- 最近一次错误。

常用查询：

```rust
runtime.active()
runtime.pending()
runtime.ready()
runtime.state()
runtime.last_error()
runtime.active_scene_id()
runtime.active_session_id()
```

### 4.4 SceneSessionId

同一个 `SceneId` 可以被多次进入，每次进入对应一个 `SceneSessionId`。清理实体、查询 spawn、标记 ready、关联 authority 时都应使用 session。

游戏层生成场景实体时必须把 `SceneOwned(session_id)` 挂到实体上，避免退出场景后残留。

## 5. 场景注册

场景进入前必须先注册到 `SceneRegistry`。

注册纯 UI 场景：

```rust
registry.register_pure_ui("ui.login")?;
```

注册首包 manifest 场景：

```rust
registry.register_manifest_scene(
    "sample.dungeon_room",
    SceneKind::Dungeon,
    "scenes/sample_dungeon_room/scene.ron",
)?;
```

注册表会校验：

- `scene_id` 不为空。
- `scene_id` 格式合法。
- 不能重复注册同一个 `scene_id`。
- manifest path 不能是空字符串。
- fallback scene 必须先注册再设置。

注意：注册 manifest 场景只会告诉 framework “这个场景有哪些资源和元数据”。具体内容实例化仍由 game layer 负责。

## 6. 场景命令

场景命令通过 Bevy message 发送：

```rust
commands.write(SceneCommand::Enter(SceneEnterRequest::new("sample.dungeon_room")));
```

当前命令：

- `Enter(SceneEnterRequest)`：进入指定场景。
- `Exit(SceneExitRequest)`：退出当前或指定场景。
- `Switch(SceneSwitchRequest)`：退出当前场景并进入新场景。
- `MarkReady(SceneReadyCommand)`：外部系统标记场景 ready。
- `Preload(ScenePreloadRequest)`：预加载命令形态已存在，当前尚未执行真实预加载。
- `Unload(SceneUnloadRequest)`：卸载命令形态已存在，当前尚未执行真实资源卸载流程。
- `ReloadCurrent(SceneReloadRequest)`：重载当前或 pending 场景。
- `SetLayerEnabled(SceneLayerCommand)`：设置 layer 启停状态。
- `SetStreamingEnabled(SceneStreamingCommand)`：启停 streaming driver。

`SceneEnterRequest` 主要字段：

- `scene_id`
- `session_id`
- `spawn_point`
- `content_version`
- `transition`
- `authority_mode`
- `seed`

单机样板场景通常只需要 `scene_id`，其余字段使用默认值。

## 7. 生命周期状态

当前生命周期状态：

```text
Idle
Resolving
Downloading
LoadingAssets
Instantiating
Activating
Active
Suspending
Deactivating
Unloading
Failed
```

当前首包 manifest 场景的典型进入流程：

1. 游戏层发送 `SceneCommand::Enter` 或 `Switch`。
2. framework 从 `SceneRegistry` 查找 `SceneDefinition`。
3. 状态进入 `Resolving`，发送 `SceneEvent::Resolving`。
4. 如果场景没有 manifest，直接按注册定义完成进入。
5. 如果场景有 manifest，从 `project/assets/` 下读取 RON。
6. 校验 manifest 版本、scene_id、layer、asset、spawn、anchor、trigger、chunk。
7. 状态进入 `LoadingAssets`。
8. 对 manifest 中声明的资源调用 `AssetServer::load_untyped`。
9. required 资源加载完成前保持 pending。
10. required 资源失败时进入 `Failed` 并发送 `SceneEvent::Failed`。
11. optional 资源失败不阻塞进入，但会记录失败计数。
12. required 资源完成后进入 `Instantiating`。
13. 如有世界内容，生成 `SceneRoot`、`SceneLayerRoot`、`SceneRuntimeRoot`。
14. 建立 spawn/anchor 索引，记录 chunk 元数据，生成 trigger。
15. 根据 camera 配置生成基础场景相机。
16. 状态进入 `Active`，发送 `SceneEvent::Entered`。
17. 本地 authority 模式下自动进入 ready；远端或外部 authority 模式下发送 ready 请求。

退出流程：

1. 游戏层发送 `SceneCommand::Exit`。
2. framework 发送 `SceneEvent::ExitStarted`。
3. 清理当前 session 的加载队列、spawn 索引、streaming 状态。
4. despawn 所有匹配 `SceneOwned(session_id)` 的实体。
5. 清空 active、pending 和 ready。
6. 状态回到 `Idle`。
7. 发送 `SceneEvent::Exited`。

## 8. 场景事件

主要事件：

- `Resolving`：开始解析场景。
- `LoadProgress`：资源加载进度变化。
- `Instantiating`：进入实例化阶段。
- `Entered`：场景进入完成，active session 已可用。
- `Ready`：场景 ready。
- `InputResetRequested`：场景切换或退出时请求重置输入。
- `ExitStarted`：开始退出。
- `Exited`：退出完成。
- `LayerLoaded` / `LayerUnloaded`：layer 状态变化。
- `ChunkLoaded` / `ChunkUnloaded`：chunk 状态变化。
- `Failed`：进入、加载或激活失败。

游戏层接入场景时，最常用的是：

- 在 `Entered` 后生成具体场景内容和 HUD。
- 在 `Exited` 或 UI 路由退出时清理业务资源。
- 在 `Failed` 后显示业务错误提示。
- 在 `InputResetRequested` 后清理按下、拖拽、蓄力等输入状态。

## 9. Manifest

framework manifest 当前使用首包 RON 文件，路径相对 `project/assets/`。

注册路径示例：

```text
scenes/sample_dungeon_room/scene.ron
```

最小结构：

```ron
(
  version: "1",
  scene_id: SceneId("sample.dungeon_room"),
  kind: "dungeon",
  entry: (
    default_spawn: Some(SceneSpawnPointId("spawn.default")),
    camera: Some((
      mode: "fixed3d",
      position: Some((0.0, 8.0, 10.0)),
      rotation: Some((-38.0, 0.0, 0.0)),
      projection: (kind: "perspective3d", fov_y: 0.9, near: 0.1, far: 200.0),
    )),
    loading_policy: "blocking_required_assets",
  ),
  layers: [
    (
      id: SceneLayerId("terrain"),
      required: true,
      assets: [
        (
          id: SceneAssetId("terrain.floor.large"),
          kind: "gltf_scene",
          path: "models/scenes/kaykit_dungeon_remastered/floor_tile_large.gltf",
        ),
      ],
    ),
  ],
  spawn_points: [
    (
      id: SceneSpawnPointId("spawn.default"),
      position: (0.0, 0.0, 0.0),
      rotation: (0.0, 0.0, 0.0),
      tags: ["default", "player"],
    ),
  ],
  anchors: [],
  triggers: [],
  chunks: [],
)
```

字段说明：

- `version`：当前固定为 `"1"`。
- `scene_id`：必须和注册时的 scene_id 一致。
- `kind`：场景类型。
- `entry.default_spawn`：默认出生点 ID。
- `entry.camera`：场景相机配置或相机引用。
- `entry.loading_policy`：Loading 策略。
- `layers`：资源分层。
- `spawn_points`：出生点。
- `anchors`：稳定挂点。
- `triggers`：触发区域。
- `chunks`：流式加载分块元数据。

当前支持的 asset kind：

- `gltf_scene`
- `gltf`
- `glb`
- `image`
- `texture`
- `audio`
- `sound`
- `ron`
- `json`
- 其他字符串会保留为 `Other`

重要限制：

- manifest 中的 glTF 资源只会被加载和跟踪，不会自动 spawn 到世界。
- game layer 必须自己根据 layout 或逻辑实例化 glTF。
- manifest 路径和 asset 路径必须是 assets 相对路径，不能包含反斜杠、绝对路径、盘符或 `..`。

## 10. 资源加载和 Loading

场景资源加载由 `SceneAssetLoadQueue` 维护。

加载行为：

- 对每个 manifest asset 调用 `AssetServer::load_untyped`。
- required layer 中的 asset 计入 required。
- optional layer 中的 asset 计入 optional。
- required 未完成前不会进入 `Active`。
- required 失败会触发 `SceneEvent::Failed`。
- optional 失败不阻塞进入。
- 资源加载进度通过 `SceneEvent::LoadProgress` 通知外部。

Loading 策略：

- `None`：不打开全局 Loading。
- `Spinner`：打开 Loading，不显示精确计数。
- `Progress`：打开 Loading，并显示加载计数。
- `Blocking`：默认策略，打开 Loading，并显示加载计数。
- `NonBlocking`：后台语义，不打开阻塞 Loading。

`sync_scene_loading_ui` 会把场景加载进度转换为 UI 框架的全局 Loading 面板命令。场景进入、退出或失败时会关闭当前 Loading。

## 11. 根实体和清理

世界内容场景会创建：

- `SceneRoot`：场景世界内容根。
- `SceneLayerRoot`：layer 根。
- `SceneRuntimeRoot`：运行时动态内容根。
- `SceneOwned`：session 归属标记。

纯 UI 场景不会创建世界 root。

游戏层实例化场景内容时必须遵守：

- 所有随场景销毁的实体都挂 `SceneOwned(session_id)`。
- 尽量把实体挂到当前场景的 root、runtime root 或 layer root 下。
- 不随场景销毁的全局实体不要挂 `SceneOwned`。
- HUD 不挂到 `SceneRoot`，它属于 UI 层，应由 UI 框架和 `AppUiMode` 管理。

退出时 framework 会 despawn 当前 session 的 `SceneOwned` 实体。因此只要游戏层正确挂标记，地形、灯光、相机、trigger 和动态对象都能随场景退出清理。

## 12. 相机

当前相机能力包括：

- `SceneCameraMode::UiOnly2d`
- `SceneCameraMode::Gameplay2d`
- `SceneCameraMode::Gameplay3d`
- `SceneCameraMode::Fixed3d`
- `SceneCameraMode::DebugFree`

manifest 可配置相机位置、旋转和投影。framework 会在场景进入时按配置生成基础相机，并挂：

- `SceneCameraRig`
- `SceneOwned(session_id)`
- `Name`

当前相机能力只覆盖基础生成和配置。相机跟随、镜头震动、镜头过渡、自由调试控制和复杂相机栈属于 game layer 或后续扩展。

注意当前项目已有全局 `Camera2d` 用于 UI/现有页面。引入 3D 样板场景时需要验证 2D 和 3D 相机不会在渲染顺序、清屏和 UI 显示上互相干扰。

## 13. Spawn 和 Anchor

manifest 中的 `spawn_points` 和 `anchors` 会被转换为 `SceneSpawnRegistry` 中的 session 索引。

游戏层可查询：

- 默认出生点。
- 指定 spawn point。
- 指定 anchor。
- 带某个 tag 的 spawn points。
- 带某个 tag 的 anchors。

适用场景：

- 玩家出生。
- 相机看向点。
- NPC 站位。
- 交互点。
- 传送门。
- 调试位置。

推荐把稳定逻辑位置写入 manifest，而不是依赖美术节点名称。

## 14. Trigger

manifest trigger 会被 framework 生成通用触发实体。当前 trigger 负责空间检测和事件派发，不处理具体业务。

framework 可发出 `SceneTriggerEvent`，游戏层再决定：

- 是否显示交互 UI。
- 是否进入传送。
- 是否触发任务。
- 是否开启战斗。
- 是否切换音乐。
- 是否启停 layer。

trigger 不应在 framework 中硬编码任务、剧情或战斗逻辑。

## 15. Streaming 和 Chunk

当前 `streaming.rs` 已有 zone、region、chunk 元数据和状态结构，也有 streaming driver 配置。

当前能力重点：

- 从 manifest 接收 chunk 元数据。
- 记录当前 session 的 chunk 状态。
- 提供启停 streaming 的命令边界。

当前尚未实现：

- 按玩家位置真实加载/卸载 chunk。
- 动态资源分块加载。
- 远离 chunk 自动释放内容。
- chunk 级别碰撞、导航和实体迁移。

小型样板场景不需要依赖 streaming。后续大世界或大型副本再接入。

## 16. Authority Ready

场景框架包含 framework 层 authority ready 边界。

本地模式：

- `SceneAuthorityMode::Local`
- `SceneAuthorityMode::LocalHost`

这两类模式可由 framework 自动标记 ready。

远端或外部模式：

- `SceneAuthorityMode::Remote`
- `SceneAuthorityMode::External`

这两类模式会发出 ready 请求，等待外部 authority 或 game layer 确认。

framework 不依赖 `project/src/game/authority/`，也不直接处理房间、玩家同步或帧同步规则。

## 17. 调试能力

当前支持从环境变量读取调试配置：

```powershell
$env:MYBEVY_START_SCENE="sample.dungeon_room"
$env:MYBEVY_START_SPAWN="spawn.default"
$env:MYBEVY_SCENE_DEBUG="true"
$env:MYBEVY_SCENE_LOG_LIFECYCLE="true"
$env:MYBEVY_SCENE_SLOW_LOADING_SECONDS="1.5"
$env:MYBEVY_SCENE_SIMULATE_FAILURE="asset_load"
cargo run
```

常用用途：

- 启动后自动进入某个已注册场景。
- 指定启动 spawn。
- 打开生命周期日志。
- 模拟慢加载。
- 模拟 manifest 或 asset 加载失败。

注意：启动场景仍必须先由 game layer 注册，否则会进入 `SceneNotFound` 失败。

## 18. 错误处理

场景失败会记录到 `SceneRuntime.last_error`，并发送 `SceneEvent::Failed`。

常见失败：

- 场景未注册。
- manifest 不存在。
- manifest 解析失败。
- manifest 版本不支持。
- manifest scene_id 和注册 scene_id 不一致。
- 默认 spawn 不存在。
- required asset 缺失或加载失败。
- 相机或实例化阶段失败。

面向玩家的文案应由 UI 和 i18n 处理，不应直接展示内部路径、堆栈或原始错误给玩家。

## 19. 当前明确未完成项

以下能力当前不应当被业务代码假设为已完成：

- 真实后续下载和 `content_cache://` 加载。
- CDN 内容版本校验和哈希校验。
- 自动实例化 manifest 中声明的 glTF。
- 复杂 additive scene 合并。
- 自动碰撞体生成。
- 自动导航网格生成。
- 自动音频区域和光照环境应用。
- 基于位置的真实 chunk 加载/卸载。
- 可视化场景调试面板。
- 完整联机场景 ready 汇总和远端 authority gating。

## 20. Framework 使用契约

游戏层使用 scene framework 时应遵守：

- 先注册场景，再发送进入命令。
- 不在 framework 层写具体游戏业务。
- manifest 写资源和元数据，不写复杂关卡脚本。
- 具体内容实例化放在 `project/src/game/scenes/`。
- HUD 放在 `project/src/game/screens/gameplay/`。
- 所有随场景销毁的实体挂 `SceneOwned(session_id)`。
- 使用 `SceneEvent::Entered` 作为开始生成具体场景内容的主要时机。
- 使用 `SceneEvent::Failed` 处理错误回退。
- 使用 `SceneEvent::InputResetRequested` 清理旧输入状态。
- 不把后续下载资源放入 `project/assets/`。

## 21. 基础验收

修改 framework scene 能力后至少执行：

```powershell
Set-Location project
cargo fmt --check
cargo check
cargo test
Set-Location ..
git diff --check
```

如果修改了场景生命周期、加载、清理或 root 逻辑，还应增加重复进入/退出测试，确认 `SceneOwned` 实体不会残留。
