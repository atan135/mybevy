# 服务端最新登录流程客户端验收

本文记录 mybevy 客户端接入 MyServer 最新账号登录、选角、character-bound ticket、game proxy 鉴权、进房、断线重连和四属性链路的验收步骤。对应服务端仓库默认位于 `C:\project\MyServer`。

## 身份口径

- `player_id` 是账号级身份，只用于登录、账号安全、封禁、kick、ticket version 和服务端审计。
- `character_id` 是玩法主体。房间、匹配、输入、移动、战斗、背包、transfer、Room snapshot、Frame input、authority local subject 和 Robot Sync/Touch Ripple replay 都必须使用当前 ticket 绑定的 `character_id`。
- `AuthRes.player_id` 仍保存在 `MyServerSession.player_id`，但客户端发给 authority/玩法层的 `MyServerEvent::Authenticated.player_id` 值是当前 `character_id`。这里的字段名沿用 authority 抽象里的 player 术语，不表示账号 `player_id`。
- 缺少 `character_id` 的 legacy ticket、ticket 与当前选角不一致、或 `AuthRes ok=true` 但本地没有选中角色，均按 `MISSING_CHARACTER_ID` / `TICKET_CHARACTER_MISMATCH` 可观察失败处理，不新增 `player_id` fallback。

## 本地栈启动

服务端完整栈：

```powershell
Set-Location C:\project\MyServer
powershell -ExecutionPolicy Bypass -File .\scripts\dev-stack.ps1 -Status
powershell -ExecutionPolicy Bypass -File .\scripts\dev-stack.ps1 -Restart -WithMatch
```

常用端口：

- `auth-http`: `http://127.0.0.1:3000`
- `game-proxy`: KCP `127.0.0.1:4000`，TCP fallback `127.0.0.1:14000`
- `game-server`: `127.0.0.1:7000`，admin `127.0.0.1:7500`
- Redis `6379`、NATS `4222`、PostgreSQL `5432`

联调前先确认没有 dev-stack PID 文件缺失但端口仍被旧进程占用的半启动状态。若存在旧 `game-proxy.exe` 或其他服务监听目标端口，先由主控确认是否停止，worker 不应擅自杀掉他人进程。

## 登录到鉴权闭环

正式账号路径：

1. 如需测试账号，先在服务端执行 `powershell -ExecutionPolicy Bypass -File .\scripts\seed-auth-test-accounts.ps1`。
2. 客户端登录页输入账号密码，例如 `test001 / Passw0rd!`，发送 `POST /api/v1/auth/login`。
3. 客户端收到 access token 后拉取 `GET /api/v1/characters`。
4. 若列表为空，创建角色 `POST /api/v1/characters`；若已有角色，读取 `GET /api/v1/characters/{character_id}/profile`。
5. 选择角色，发送 `POST /api/v1/characters/select`，请求体只包含 `character_id`。
6. 客户端保存 `player_id`、当前 `character_id`、`world_id`、角色资料、ticket 和 ticket 过期时间。
7. 使用 character-bound ticket 连接 game proxy，发送 `AuthReq`。
8. 收到 `AuthRes ok=true` 后，session 保存账号级 `AuthRes.player_id`，authority/玩法层使用 ticket 中的当前 `character_id` 作为本地主体。
9. 验证日志只出现 `account_player_id`、`character_id`、`ticket_fp`、`ticket_remaining_seconds`，不出现 access token、ticket 或密码明文。

游客路径：

1. 登录页点击游客登录，或设置 `MYSERVER_GUEST_ID` 后由自动入口发送 `POST /api/v1/auth/guest-login`。
2. 之后的拉角色、创建角色、选择角色、签票、game proxy 鉴权流程与正式账号一致。
3. 游客升级、登录态持久化和安全存储仍是暂缓项，不作为本轮验收完成条件。

## 进房与玩法输入

最小进房：

1. `MyServerEvent::Authenticated` 发出后，Touch Ripple、Robot Sync 或 authority dev 入口发送 `JoinRoom`。
2. `RoomJoined(ok=true)` 后发送 `SetReady(true)`。
3. `ReadyChanged(ok=true)` 后发送 `StartRoom`。
4. 服务端推送 `RoomStatePush`，客户端 snapshot 的成员列表来自 `RoomMember.character_id`。
5. 玩法输入通过 `AuthorityCommand::SendInput` 进入 MyServer endpoint；客户端不在请求中带账号级 `player_id`，服务端按 ticket/session 绑定角色落主体。
6. `FrameBundlePush` 中的 `FrameInput.character_id` 被映射到 authority `PlayerInput.player_id`，Touch Ripple 和 Robot Sync replay 以该值区分实体。

建议分别验收：

- `ui_touch_room`: 按下、拖动、松开后两端能看到同一帧序的硬边圆和水波拖尾。
- `robot_sync_room`: 两端进入 `arena.robot_sync`，本地机器人移动由 authority frame 推进，Robot Sync HUD 的 player/local subject 为 `chr_*`。
- `AUTHORITY_DEV_MODE=myserver`: 自动登录、进房、ready、start、定时输入仍可串联。

## ticket、断线和服务端控制

需要观察的路径：

- 提前补发：ticket 剩余时间小于 `MYSERVER_TICKET_REFRESH_MARGIN_MS` 时发送 `/api/v1/game-ticket/issue`，请求体必须是当前 `character_id`。
- 断线重连：启用 `MYSERVER_AUTO_RECONNECT_WITH_FRESH_TICKET=true` 后，断线先补发 ticket，再重新 AuthReq。
- reconnect cursor：重连发送 `RoomReconnectReq.last_character_push_sequence`，值来自本地 `CharacterElementsCache.last_push_sequence`。
- redirect：收到服务端 redirect 后，客户端补发 ticket、连接新 endpoint、AuthReq 成功后发送 `RoomReconnectReq`，不发普通 `Authenticated` 进房事件。
- kick：收到 kick/revoke/duplicate login/ban/maintenance 等推送后，客户端清理连接和待处理 HTTP，请求层暴露 `AccountStatusChanged` 或 `GameAuthRejected`，不得自动用旧 ticket 重连。
- 维护模式：`MAINTENANCE`、`SERVER_MAINTENANCE` 或 HTTP 维护状态应显示为可诊断失败，保留 request id、HTTP status、error code 和 endpoint。

## 四属性验收

资料与快照：

1. 选角或 profile 返回中若带 `attributes`，客户端写入 `CharacterElementsCache`，但不把 profile 当作永久预测源。
2. 进入游戏后发送 `GetCharacterElementsReq(1413)`，收到 `GetCharacterElementsRes(1414)` 后刷新当前 `character_id` 的 `affinity`、`mastery` 和 `snapshot_refreshed_at`。
3. UI/调试面板展示地、火、水、风四属性时，以 `CharacterElementsCache` 当前快照为准。

push 与重连：

1. `CharacterElementsChangePush(1505)` 到达后，客户端只消费服务端 `after` 快照，更新 `last_push_sequence` 和 `last_push_revision`。
2. 过期、重复或乱序 push 的去重/保序由服务端和 reconnect cursor 共同兜底；客户端断线后必须重拉 `GetCharacterElementsReq`，不要本地永久累加 delta。
3. 断线重连成功后，客户端发送 `RoomReconnectReq.last_character_push_sequence`，随后发送 `GetCharacterElementsReq` 获取权威快照。
4. 道具、战斗、任务或场景临时效果造成的永久变化，客户端只能在响应或 push 到达后展示；临时 `effective_elements` 不写入永久 `affinity` / `mastery`。

## 日志与诊断字段

允许记录：

- HTTP: request id、operation、endpoint path、HTTP status、服务端 error code、状态转换。
- game proxy: connection id、seq、message type、endpoint、transport、错误码。
- session: account `player_id`、当前 `character_id`、`world_id`、状态机阶段。
- ticket: `ticket_fp`、payload fingerprint、过期时间、剩余秒数。
- reconnect: cause、redirect target、room id、rollout epoch、last push sequence。

禁止记录：

- access token、refresh token、game ticket 原文。
- 密码、Authorization header、Cookie、完整请求体中包含凭据的字段。
- 可复用的 ticket payload 原文或未脱敏 JWT。

脱敏规则：

- 凭据只记录短指纹，当前客户端使用 `DIAGNOSTIC_FINGERPRINT_LEN = 12` 的稳定短 fingerprint。
- 本地需要更密集的状态日志时设置 `MYSERVER_DIAGNOSTIC_TRACE=true`；线上默认保持 trace/debug 级别，不应把凭据相关字段提升到 info。
- 线上串联问题时按 request id、connection id、seq、endpoint、error code、ticket_fp、character_id、状态转换联合定位。
